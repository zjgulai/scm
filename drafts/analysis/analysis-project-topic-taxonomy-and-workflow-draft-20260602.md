# 项目主题与专题治理工作流盘点草稿

## 1. 当前判断

当前项目的宏观规划不是“多个资料文件夹并列”，而是一个业务决策产品体系：

```text
4 大业务专题
  -> 14 个子课题
  -> 4 条交叉故事线
  -> 5+1 智能体
  -> 全局数据契约
  -> Phase2 / Phase3 管道与运营化
```

供应链分枝已经完成的工作流，不应机械复制到每个目录。正确做法是先识别主题层级，再按主题独立建立产品化链路，避免把支撑资料、样例产物和正式专题混成一层。

## 5. 早期样例专题产物的地位

`main_project_lute/data_example/专题产物/` 不是宏观规划里的 4 大专题本体。它更像早期分析样例和脚本产物库：

| 样例目录 | 文件数 | 初步判断 | 是否直接套供应链工作流 |
|---|---:|---|---|

| `专题01-sheet10` | 19 | 用户多维洞察扩展 | 否，先映射到专题④ LTV 或用户精准营销 |
| `专题02` | 7 | 双平台费率归因 | 否，先映射到专题②成本结构 |

| `专题04` | 12 | 订单价、订单数、毛利率归因 | 否，先映射到专题②子课题③ |

## 6. 供应链分枝定位

`scm/` 当前定位是供应链专题库。它与专题②高度相关，但比专题②更细，聚焦供应链成本效率、履约节点、库存、供应商、逆向物流和动作闭环。

已完成的供应链工作流包括：

```text
SCM 蓝图
  -> Data 层重建
  -> Plan / BI PRD
  -> Agent 任务规格
  -> Runtime 原型路由
  -> Source 确认矩阵
  -> Source 确认包
  -> DQ 规格
  -> SQL 前置规格
```

这个工作流可以作为其他主专题的治理模板，但前缀和边界必须独立：

| 主专题 | 建议前缀 | 不应交叉的内容 |
|---|---|---|
| 专题① VOC | `VOC-*` | 不直接引用订单根因，除非进入交叉线 |
| 专题② 订单/成本/退款 | `ORDER-*` 或 `ORD-*` | 不把供应链专项全部吞入主专题 |
| 专题③ 渠道 | `CHANNEL-*` 或 `CH-*` | 不把营销 ROI 结论当渠道健康结论 |
| 专题④ 营销 ROI | `MKT-*` | 不把 VOC 高浓度社区直接等同 ROI 结果 |
| 交叉线 | `XL1-*`、`XL2-*`、`XL3-*`、`XL4-*` | 只消费上游结果，不重建上游专题 |
| 供应链 | `SCM-*` | 保持独立专题包 |

## 7. 建议复制的标准工作流

每个主专题都应独立执行以下链路：

```text
TOPIC-BLUEPRINT-001  分层蓝图
TOPIC-DATA-001       指标字典 / 指标种子表
TOPIC-DATA-002+      主题宽表规格
TOPIC-BI-001+        看板 / 分析视图 PRD
TOPIC-AGENT-001+     Agent 任务规格
TOPIC-RUNTIME-001    Data Agent 路由与输出边界
TOPIC-SOURCE-001     本地证据与真实源表确认矩阵
TOPIC-SOURCE-002     真实源系统确认包
TOPIC-DQ-001         样本质量校验规格
TOPIC-SQL-001        SQL 初稿前置规格
```

交叉线不走完整主专题链路。交叉线应执行：

```text
XL-BLUEPRINT-001     编排蓝图
XL-IO-001            上游输出与下游输入契约
XL-AGENT-001         参与 Agent 与触发条件
XL-DQ-001            上游结果表质量门槛
XL-RUNTIME-001       编排路由与禁止越界规则
```

## 8. 推荐执行顺序

| 顺序 | 对象 | 原因 | 预期产出 |
|---:|---|---|---|
| 1 | 专题② 线上订单数量与质量提升 | Phase2 和供应链分枝资产最多，且交叉线3依赖它 | `ORDER-*` 产品化蓝图与 Data / Agent / Source / DQ / SQL 前置规格 |
| 2 | 交叉线3 订单与退款反哺 VOC | 与专题②强耦合，业务价值直接 | `XL3-*` 编排蓝图与 IO 契约 |
| 3 | 专题① 全域 VOC 数据洞察 | 是交叉线1/2/3/4的核心上游 | `VOC-*` 产品化蓝图 |
| 4 | 专题③ 渠道健康 | 已有 Phase3 output，依赖较清楚 | `CHANNEL-*` 产品化蓝图 |
| 5 | 专题④ 营销 ROI | 依赖活动分类、曝光和 ROI 归因口径，决策点较多 | `MKT-*` 产品化蓝图 |
| 6 | 交叉线1/2/4 | 等上游专题产出稳定后治理 | `XL1/XL2/XL4-*` 编排规格 |

## 9. 需要你决策的问题

### D1. 供应链是否正式定义为专题②的独立子分枝

推荐：保留 `scm/` 顶层专题库，同时在专题②蓝图中把 SCM 标为“深挖分枝”，不迁入 `main_project_lute/`。

理由：供应链已经有独立图谱和 18 个 Markdown，如果强行并入专题②，会让专题②蓝图过重。

### D2. 是否先治理专题②，而不是四个专题同时铺开

推荐：先专题②，再交叉线3，再专题①。

理由：专题②已有 Phase2 输出、订单/退款数仓契约、供应链深挖分枝，最接近供应链工作流的成熟度。

### D3. 早期 `data_example/专题产物/专题01-05` 是否要重新映射回 4 大专题

推荐：需要映射，但不要把它们提升为主专题。

建议方式：建一张“样例产物 -> 四大专题/子课题/可复用能力”映射表，作为后续产品化证据，不改变目录名。

### D4. 是否为四大专题建立独立正式蓝图文件

推荐：先在 `drafts/analysis/` 完成控制面板，经你确认后，再在 `main_project_lute/Phase3_全专题与运营化/` 或各专题目录下转正式。

原因：当前还没有独立的 `topic1_voc/`、`topic2_order/` 等正式专题治理目录，贸然新增会改变项目结构。

### D5. 命名前缀

推荐：

| 对象 | 前缀 |
|---|---|
| 专题① | `VOC-*` |
| 专题② | `ORDER-*` |
| 专题③ | `CHANNEL-*` |
| 专题④ | `MKT-*` |
| 交叉线 | `XL1-*` 到 `XL4-*` |
| 供应链 | 保持 `SCM-*` |

## 10. 下一步建议

若你同意以上边界，下一步不应继续动供应链，而应执行：

```text
ORDER-BLUEPRINT-001
  -> 以专题②为对象
  -> 独立识别订单/成本/退款/仓网四个子课题
  -> 标注供应链 SCM 为专题②的深挖分枝
  -> 对齐 Phase2 output、data_example 专题02/04/05、全局数仓 01/05
  -> 形成专题②产品化拆分与数据任务蓝图
```

本步骤仍不会写正式 SQL。真实 SQL 必须等待真实源表、样本和 DQ 通过。
