# 主体项目架构与「卓越商业分析专家AI SaaS」下一步计划

> 基于对 `main_project_lute` 的探索，总结当前架构、与 SaaS 目标的差距，以及建议的下一步执行计划。  
> 主体项目：main_project_lute；data_example 为算法/数据测试；cross_border_ecommerce_skills、mckinsey_ppt_skills、ref 等为其服务。

## 一、main_project_lute 当前架构总览

### 1.1 定位与范围

| 维度 | 内容 |
|------|------|
| **产品定位** | 数智决策-AI效能作战室（母婴出海场景，60～80 亿规模） |
| **业务范围** | 4 大专题 × 14 子课题；4 条交叉分析故事线 |
| **智能体** | 5+1：VOC/消费者、订单/成本/退款/仓网、渠道/国家、营销/LTV/ROI、战略解码/业财、咨询PPT/故事线 |

## 二、与「卓越商业分析专家AI SaaS」的差距

当前项目是**单企业、单场景（母婴出海）**的「数智作战室」；要做成**对外交付的「卓越商业分析专家AI SaaS」**，需在以下维度补足或明确路径。

| 维度 | 当前 | SaaS 目标 | 差距 |
|------|------|-----------|------|
| **租户与隔离** | 单企业（Momcozy） | 多租户、数据/权限隔离 | 需租户模型、数据隔离与权限设计 |
| **行业与场景** | 母婴出海、4 专题 | 可配置专题/行业模板或通用分析能力 | 专题与行业「可配置化」或「模板化」 |
| **产品形态** | 文档+脚本+Agent 设计 | 可用的产品界面（Web/API）、工作流可触发 | 需要应用层：前端/API/工作流引擎 |
| **分析能力交付** | Skills + 故事线 + data_example 脚本 | 标准化分析流水线 + 可选扩展 | 分析能力需「服务化」「可编排」 |
| **计费与商业化** | 无 | 按租户/用量/专题的计费与配额 | 需计费模型与用量统计 |
| **接入方式** | 本地/内网数据与脚本 | 数据接入（上传/API/连接器）、安全与合规 | 需数据接入层与安全规范 |

## 三、建议的下一步执行计划

在**不放弃现有 Phase 0～3 节奏**的前提下，分两条线推进：  
- **A 线**：继续把「数智作战室」做闭环（为 SaaS 提供可复用的分析内核）。  
- **B 线**：为「卓越商业分析专家AI SaaS」做产品化与架构准备。

### 3.1 A 线：优先完成闭环（建议 1～2 个月内）

| 序号 | 工作项 | 产出 | 说明 |
|------|--------|------|------|
| A1 | Phase 0 收尾 | ref/data_index 或指标字典索引；ref/roles_skills 下 5+1 Agent 初版；与数仓的数据契约确认 | 与《02_接下来开发计划与流程》P0-1、P0-2 一致 |
| A2 | Phase 1 收尾 | 专题故事线「可执行模板」；5+1 Agent 与 Skill 映射表；4 条交叉线编排图 | P1-1～P1-3 |
| A3 | Phase 2 MVP 落地 | run_phase2_topic2_pipeline.py 打通专题②；run_phase2_crossline3_pipeline.py 打通退款→VOC；产出 topic2_* 表 + refund_theme_input_for_voc | 与《Phase2_MVP_开发/00_Phase2_MVP_实施清单》对齐 |
| A4 | 一条「提问→分析→PPT」演示链 | 从选定问题 → 调用 Agent/脚本 → 生成 2～3 页 PPT 或 Markdown 结论 | 验证端到端可交付，为 SaaS 演示打样 |

### 3.2 B 线：SaaS 化准备（与 A 线并行，先做设计与拆解）

| 序号 | 工作项 | 产出 | 说明 |
|------|--------|------|------|
| B1 | SaaS 产品愿景与范围文档 | 一页纸：目标用户、核心价值、首期功能范围、与现有 4 专题的关系（保留/抽象/模板） | 明确「卓越商业分析专家」的边界与首版范围 |
| B2 | 多租户与数据隔离设计 | 租户模型（租户 ID、数据隔离策略）、权限模型（角色/资源）初版 | 为后续账号体系与数据安全打基础 |
| B3 | 能力抽象：分析服务层 | 将「专题→故事线→Agent/Skill→输出」抽象为可调用的「分析任务」或「分析 API」清单（名称、输入、输出、依赖数据） | 便于后续封装为 API 或工作流节点 |
| B4 | 应用形态选择 | 文档：首版是「API 优先」还是「Web 控制台优先」、是否与 Cursor/Claude Code 深度集成 | 决定先做 API 还是先做 UI |

### 3.3 建议执行顺序（时间线）

| 时间段 | 执行内容 | 产出 |
|--------|----------|------|
| **第 1～2 周** | A1（Phase 0 收尾）+ B1（SaaS 愿景与范围） | ref 骨架与数据契约；SaaS 一页纸范围 |
| **第 3～4 周** | A2（Phase 1 收尾）+ B2（多租户与隔离设计） | 故事线模板与交叉线编排；租户/权限设计初版 |
| **第 5～8 周** | A3（Phase 2 MVP）+ B3（分析服务层抽象） | 专题② + 交叉线3 跑通；分析任务/API 清单 |
| **第 8 周后** | A4（演示链）+ B4（应用形态） | 一条端到端演示；API vs Web 决策与首版路线图 |

## 五、结论与下一步动作

- **架构结论**：main_project_lute 已具备清晰的「4 专题 + 5+1 Agent + 数据层」架构与 Phase 0～3 路线；数据资源整合已交付数仓，故事线与交叉线文档齐全，Phase 2 MVP 有明确实施清单，Phase 3 沙盘与 I/O 已就绪。
- **SaaS 差距**：需补足多租户与数据隔离、分析能力服务化、产品形态（API/Web）、计费与接入方式等；建议用 B 线在不大幅打乱 A 线的前提下逐步补足。
- **建议下一步**：  
  1. **立即**：完成 Phase 0 收尾（ref 骨架 + 数据契约）并落笔 B1（SaaS 愿景与首期范围）。  
  2. **随后**：按《02_接下来开发计划与流程》推进 Phase 1 收尾，并启动 run_phase2_topic2_pipeline / run_phase2_crossline3_pipeline 实现。  
  3. **并行**：推进 B2～B4，在 MVP 跑通时形成「分析服务层」抽象与首版应用形态选择，为「卓越商业分析专家AI SaaS」定下可执行路线图。
