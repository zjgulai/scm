# 沙盘推演计划

> 用虚拟数据验证「数据 → 专题 → 交叉线 → 战略/PPT」全链路可跑通、I/O 无断点；为接真数做准备。

## 一、目标

- **模块级**：在暂用虚拟数据的前提下，跑通各专题管道（或管道骨架）与交叉线节点，验证输入输出字段与《模块输入输出规格表》一致。
- **沙盘级**：每次跑批后产出《沙盘评估报告》，记录本批 mock 表清单、跑通的模块、未跑通原因、I/O 缺口，便于迭代直至主要链路闭环。

## 三、生成方式

- **脚本**：扩展 `data_example/scripts/generate_phase2_mock_data.py` 或新建 `generate_phase3_sandbox_mock.py`，按上表顺序生成；每表至少包含「关键字段」子集，数据类型与《专题×交叉线 字段口径说明》一致。
- **输出目录**：统一为 `phase3_mock/`（或沿用 `phase2_mock/` 并扩展），与 Phase2 的 mock 回退约定一致。

## 四、管道消费 mock 的约定

- 各 Phase3 管道（或现有 Phase2 骨架）**优先读真实中间结果**；若对应表/分区不存在，则**回退到 mock 目录**读取同名 mock 表（与 crossline3 的 mock 回退逻辑一致）。
- 配置项建议：如 `USE_MOCK_WHEN_MISSING=true` 或环境变量，管道内判断「真实路径是否有数据」，无则读 `phase3_mock/`（或 `phase2_mock/`）。

## 五、评估报告模板

每次沙盘跑批后填写《沙盘评估报告》：

| 项目 | 内容 |
|------|------|
| 跑批日期 | YYYY-MM-DD |
| 本批 mock 表清单 | 列出本批参与跑批的 mock 表名 |
| 跑通的模块 | 专题② 子课题①～④、交叉线3、专题① 子课题① 等 |
| 未跑通的模块及原因 | 如：专题③ 无脚本；专题④ 缺 fact_user_lifecycle mock |
| I/O 缺口 | 与《模块输入输出规格表》对比，列出缺失的输入字段或输出物 |
| 与规格表一致性检查 | 通过 / 部分通过 / 未通过；备注差异 |

报告可保存为 `main_project_lute/Phase3_全专题与运营化/沙盘评估报告_YYYYMMDD.md`。

## 六、执行顺序建议

1. 落盘《模块输入输出规格表》。
2. 落盘本《沙盘推演计划》与 Mock 表清单。
3. 实现或扩展 mock 生成脚本（至少覆盖专题②+③ 或 ②+① 所需表）。
4. 运行管道骨架 + mock，产出《沙盘评估报告》初版。
5. 根据报告补 I/O 缺口与 Agent/Skill 细化，迭代沙盘直至主要链路闭环。
