# 备货库存数据规划与积加 SCM 知识库差异分析

## 结论

钉钉文档《备货库存数据规划》不是单纯的积加 ERP 仓库字段说明，而是一个面向“备货计划业务库存”的产品规划口径。它把积加库存、SRM 需求池、采购订单、送货单、调拨单、FBA 货件、自研 ERP 物流跟踪表、平台同步库存、仓库归属和销售映射组合成跨平台业务库存。

现有积加 SCM 知识库更偏“ERP 页面事实、字段字典、指标血缘底稿”。它覆盖计划、采购、仓库、物流四模块，但尚未把钉钉文档里的自研 ERP、SRM、平台接口、渠道映射和业务库存公式纳入同一套指标口径。

因此下一步不是替换积加知识库，而是在现有知识库上增加一层 `planning_business_inventory` 语义层。

## 口径差异总览

| 维度 | 钉钉《备货库存数据规划》 | 现有积加 SCM 知识库 | 差异影响 |
|---|---|---|---|
| 目标 | 推动备货计划业务库存展示、筛选、导出、更新 | 建设计划、采购、仓库、物流指标底座 | 钉钉文档是产品规划层，现有知识库是数据建模层 |
| 数据范围 | 中仓、FBA、Walmart、TikTok、Shopify、三方仓、平台接口库存 | 积加帮助文档和 ERP 仓库页面为主 | 钉钉文档多出自研 ERP 与平台接口数据 |
| 库存层级 | 数据源库存 + 计划业务库存 | ODS/DWD/DWS/ADS/META 数据分层 | 需要新增业务库存语义层 |
| 指标口径 | 计划库存、采购未交库存、在途库存、可用、预占、冻结、在库、在库良品、不良品 | 计划量、采购量、交货量、在途量、良品量、次品量、可用量、预占量 | 名称相近但计算来源和状态过滤不同 |
| 粒度 | 按渠道变化：仓库+供应链 SKU、仓库+MSKU、ItemID+国家、MSKU+国家 | SKU/MSKU + 仓库/站点 + 快照日期 | 需要按平台保留不同业务主键 |
| 状态过滤 | 明确列出采购、送货单、调拨单、FBA 货件、物流跟踪状态 | 仅在知识库中提出状态标准化方向 | 钉钉文档提供了可直接落库的状态规则 |
| 默认值 | 多处字段无数据源时默认为 0 | 现有知识库强调字段缺失需标记 | 默认 0 必须区分“业务定义为 0”和“数据缺失” |
| 刷新频率 | 多处要求每小时整点，部分同步完成后实时更新 | 现有知识库要求后续导出/API 校验刷新频率 | 刷新策略需要进入元数据表 |

## 核心指标差异

| 指标 | 钉钉规划口径 | 现有知识库口径 | 处理建议 |
|---|---|---|---|
| 计划库存 | 按平台不同，来自 SRM 需求池、采购订单、调拨单待拣货/待出库等 | 计划量更多来自采购计划、装配计划和产品库存字段 | 拆成 `planning_stock_qty`，不要直接等同 `SCM-WH-003 计划量` |
| 采购未交库存 | 采购订单明细和送货单明细组合，包含在途数量、未交货数量、送货单发货数量 | 现有指标有采购量、交货量、未交量、入库量，但未形成“采购未交库存”合并字段 | 新增 `purchase_undelivered_stock_qty` |
| 在途库存 | FBA 用货件状态和调拨单状态；Walmart/TikTok 用自研 ERP 物流跟踪表状态 | 现有知识库将 ERP 单据在途、平台报告在途、货件在途视为需区分来源 | 建三套来源：`erp_transfer_in_transit`、`platform_shipment_in_transit`、`logistics_tracking_in_transit` |
| 可用库存 | 产品库存直接取积加可用量；FBA/Walmart/TikTok 按平台字段或平台接口映射 | 现有知识库定义为可用于销售、出库或分配的库存数量 | 保留同名指标，但必须按 `inventory_source_type` 分公式 |
| 预占库存 | 产品库存取已处理预占；FBA 取买家订单预留；Walmart 默认为 0；TikTok 由 FBT 与三方仓汇总 | 现有知识库定义为订单、调拨、出库或三方仓占用数量 | 增加平台特化字段映射和默认 0 规则 |
| 冻结库存 | 产品库存、Walmart、TikTok 多处默认为 0；FBA 来自运营中心转运和正在处理预留 | 现有知识库尚未把冻结库存列为核心实采指标 | 新增冻结库存指标，但字段状态必须标记为平台依赖 |
| 在库库存 | 业务字段组合：可用 + 预占 + 冻结 + 不良品 | 现有知识库一般以在库量、良品量、次品量拆分 | 建议将其定义为业务派生指标，不作为源字段 |
| 在库良品库存 | 业务字段组合：可用 + 预占 + 冻结 | 现有知识库定义为良品库存或良品可用量 | 需要区分“良品库存”和“良品可用” |
| 不良品库存 | 产品库存取次品量；FBA 取不可售；Walmart 取拒收损坏；TikTok 汇总 FBT 次品和三方仓次品 | 现有知识库定义为次品或不良品库存 | 名称统一为 `defective_qty`，保留平台字段来源 |

## 数据源差异

| 数据源 | 钉钉规划是否使用 | 现有知识库是否覆盖 | 差异 |
|---|---|---|---|
| 积加产品库存 | 使用 | 覆盖 | 需补“已处理预占”“次品量”等业务字段映射 |
| 积加 FBA/平台仓库存 | 使用 | 覆盖 | 需补 FBA 字段公式：本地可售、预留、不可售 |
| 积加三方仓库存 | 使用 | 覆盖 | 需补渠道归属和供应链 SKU 映射 |
| SRM 需求池 | 使用 | 未覆盖 | 需新增 SRM 需求事实表 |
| SRM 采购订单 | 使用 | 采购模块有 PO 概念，但非 SRM 明细口径 | 需确认 SRM 与积加采购订单是否同源或双系统 |
| SRM 送货单 | 使用 | 交货单/入库单有类似链路 | 需新增送货单明细状态规则 |
| 调拨单 | 使用 | 覆盖 | 需区分调拨单用于库存流水、计划库存、在途库存三种场景 |
| FBA 货件 | 使用 | 物流/平台仓有货件概念 | 需补 FBA 货件状态枚举和发货量规则 |
| 自研 ERP 物流跟踪表 | 使用 | 未覆盖 | 需新增物流跟踪事实表 |
| 自研 ERP 仓库归属 | 使用 | 未覆盖 | 需新增仓库归属维表 |
| 自研 ERP 销售映射 | 使用 | 未覆盖 | 需新增 SKU/MSKU/ItemID/GTIN/FNSKU 映射表 |
| 平台接口库存/FBT 库存 | 使用 | 未覆盖 | 需新增平台接口库存快照 |

## 平台粒度差异

| 库存类型 | 钉钉规划粒度 | 现有知识库粒度 | 处理建议 |
|---|---|---|---|
| 产品库存 | 仓库 + 供应链 SKU | SKU/MSKU + 仓库/站点 + 快照日期 | 以供应链 SKU 为主键补齐映射 |
| FBA 库存 | 仓库 + MSKU | ASIN/MSKU + 平台仓 + 货件 | 保留 MSKU + 仓库，同时关联 ASIN/FNSKU |
| Walmart 库存 | ItemID + 国家 | 平台站点 + SKU/MSKU | 新增 `dim_walmart_item_mapping` |
| TikTok 库存 | MSKU + 国家，部分从供应链 SKU + 国家映射 | 平台站点 + SKU/MSKU | 新增销售映射和国家维度 |
| 三方仓库存 | 三方仓 SKU + 仓库，部分业务库存按供应链 SKU + 仓库 | 三方仓 SKU + 仓库 + 对账日期 | 同时保留三方仓 SKU 和供应链 SKU |
| Shopify/独立站库存 | 供应链 SKU + 仓库 | 当前未单独建渠道业务库存 | 新增渠道归属维度 |

## 状态规则差异

| 规则域 | 钉钉规划明确规则 | 现有知识库状态 | 缺口 |
|---|---|---|---|
| 采购计划/计划库存 | 需求池待订单；采购订单新建、变更中、待供应商确认、供应商拒绝、易仓处理中 | 只定义计划/采购/交货链路 | 需新增 SRM 状态枚举 |
| 采购未交 | 采购订单已确认；送货单新建、已拒绝、待计划确认；过滤 PO 已关闭 | 有采购订单、交货单、入库单指标 | 需补送货单状态和 PO 关闭排除 |
| FBA 在途 | FBA 货件状态包含 WORKING、SHIPPED、IN_TRANSIT、DELIVERED、CHECKED_IN、READY_TO_SHIP，且关联调拨单为调拨在途/已完成 | 仅提醒 FBA 在途来源需区分 | 可直接转成 `dim_status_code` |
| 调拨计划 | 调拨单待拣货、待出库 | 已覆盖调拨状态页签 | 需把该状态用于计划库存，而不是只用于仓库作业 |
| 物流跟踪在途 | 正常在途、出口查验未放行、进口查验未放行、延误风险 | 未覆盖自研物流跟踪表 | 新增物流跟踪状态域 |

## 对现有知识库的修正建议

### 1. 新增业务库存语义层

建议在数据库设计中增加：

| 表 | 粒度 | 用途 |
|---|---|---|
| `fact_planning_business_inventory_snapshot` | 渠道 + 业务主键 + 仓库/国家 + 快照时间 | 承载备货计划业务库存 |
| `fact_srm_demand_pool_line` | 需求池单号 + 物料编码 + 供应商 | 计划库存来源之一 |
| `fact_srm_purchase_order_line` | SRM PO + 物料编码 + 供应商 | 计划库存、采购未交库存 |
| `fact_srm_delivery_notice_line` | 送货单 + PO + SKU | 采购未交库存 |
| `fact_platform_inventory_snapshot` | 平台 + 平台业务主键 + 国家/仓库 + 快照时间 | FBA/Walmart/TikTok/FBT 平台库存 |
| `fact_logistics_tracking_line` | 物流跟踪单 + SKU 明细 | Walmart/TikTok 在途库存 |
| `dim_channel_warehouse_attribution` | 仓库 + 渠道 + 国家 + 仓库类型 | Walmart/TikTok/Shopify 仓库筛选 |
| `dim_platform_sku_mapping` | 平台 + SKU/MSKU/FNSKU/GTIN/ItemID + 国家 | 业务主键映射 |

### 2. 修改指标字典的库存派生关系

现有 `warehouse-live-metric-dictionary-draft-20260604.md` 中的派生关系偏通用，应补充业务库存公式：

```text
业务在库库存 = 可用库存 + 预占库存 + 冻结库存 + 不良品库存
业务在库良品库存 = 可用库存 + 预占库存 + 冻结库存
```

这两个公式只能用于备货业务库存，不应覆盖 ERP 原生在库量。

### 3. 增加字段证据状态

钉钉规划中的“默认为 0”需要单独状态，不能和缺失字段混同：

| 状态 | 用途 |
|---|---|
| `business-default-zero` | 业务规则明确要求默认为 0 |
| `source-missing` | 当前没有发现数据源 |
| `platform-derived` | 由平台字段组合计算 |
| `mapping-expanded` | 由一个 SKU 映射多个平台主键，存在放大风险 |

### 4. 增加映射风险规则

Walmart 和 TikTok 都存在一对多映射风险：

| 场景 | 风险 | 质量规则 |
|---|---|---|
| GTIN 对多个 ItemID | ItemID 计划库存虚高 | 记录映射基数，超过 1 时标记 |
| 供应链 SKU + 国家匹配多个 MSKU | 每条 MSKU 都拿到汇总库存 | 记录映射扩展倍数，BI 中提示 |
| 无 GTIN 或无销售映射 | 计划库存不匹配 | 落入未匹配明细表 |

## 差异分级

| 等级 | 差异 | 处理优先级 |
|---|---|---|
| P0 | 计划库存、在途库存、采购未交库存的来源和状态过滤与现有知识库不一致 | 必须补入指标字典 |
| P0 | SRM、物流跟踪表、仓库归属、销售映射未在现有知识库建模 | 必须补入数据模型 |
| P1 | Walmart/TikTok/FBA 的业务主键和默认值规则不同 | 必须补入字段映射和质量规则 |
| P1 | 同名库存字段按平台公式不同 | 必须建立 `source_system + inventory_source_type` |
| P2 | 搜索、批量搜索、导出文件名、排序筛选等页面交互规则 | 进入产品需求或 BI 导出规则，不进入核心事实表 |
