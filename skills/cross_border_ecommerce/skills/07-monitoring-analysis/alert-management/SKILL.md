# 跨境电商预警分级管理 Skill

## 概述

预警分级管理是业务监控体系的核心。本Skill提供系统化的预警管理方法，帮助用户：

- **分级管理**: 按严重程度划分P0-P3四级预警
- **标准化告警**: 统一告警卡片格式和内容
- **活动感知**: 维护活动日历和白名单
- **闭环管理**: 明确SLA标准和关闭流程

### 预警分级体系

| 等级 | 名称 | 严重程度 | 典型场景 | 响应要求 |
|------|------|---------|---------|---------|
| P0 | 紧急 | 极高 | 毛利额<0、毛利率骤降超阈 | 1小时内确认，4小时内处置 |
| P1 | 高 | 高 | GMV日环比异常>30%、退款率异常 | 4小时内确认，24小时内处置 |
| P2 | 中 | 中 | 广告费率异常、折扣率异常 | 24小时内确认，72小时内处置 |
| P3 | 低 | 低 | 轻微波动、数据质量问题 | 1个工作日内确认 |

## 核心功能

### 1. 预警分级标准

#### P0（紧急）

**触发条件**:
```
1. 毛利额 < 0（绝对红线）
2. 毛利率骤降超阈（如从40%降到10%以下）
3. GMV下降且毛利为负
4. 单SKU/单承运商退款集中且金额大
5. ODR考核红线触发（>1%）
```

**响应动作**:
```
1. 立即短信+电话通知
2. 自动创建紧急工单
3. 暂停相关促销/投放
4. 升级至管理层
```

**关闭条件**:
- 根因已定位
- 处置措施已执行并验证有效
- 或经评估为误报

#### P1（高）

**触发条件**:
```
1. GMV日环比异常 >30% 且伴随退款率上升
2. 毛利率 <35%（行业基准线）
3. 退款率 >5% 或周同比上升 >50%
4. 营业收入下降 >15% 且毛利为负
5. S-ESD检测到显著异常
```

**响应动作**:
```
1. 邮件+即时通讯通知
2. 自动拉取明细数据
3. 生成差异归因报告
4. 通知业务负责人
```

**关闭条件**:
- 根因已定位且处置措施已执行
- 或确认为误报

#### P2（中）

**触发条件**:
```
1. 广告费率异常（7日平均上升>30%且ROAS降>20%）
2. 折扣率异常（非活动期上升>15pp）
3. 毛利率 <45%（低于目标）
4. 履约成本异常上升
```

**响应动作**:
```
1. 即时通讯通知
2. 自动生成KPI影响评估
3. 通知相关部门
```

**关闭条件**:
- 根因已定位、处置措施已执行
- 或确认为误报

#### P3（低）

**触发条件**:
```
1. 轻微波动（<2σ）
2. 单点数据质量问题
3. 非核心指标偏离
```

**响应动作**:
```
1. 记录在日报中
2. 不触发即时通知
```

**关闭条件**:
- 问题已记录或已观察确认无需处置

### 2. 告警卡片模板

**标准告警卡片结构**:

```markdown
## [P0] 销售额异常下降 - 紧急告警

### 摘要行
- **指标**: 销售额(GMV)
- **等级**: P0（紧急）
- **触发时间**: 2024-03-15 10:30:00
- **影响范围**: 美国站 / 全品类

### 关键数值
| 指标 | 实际值 | 目标值 | 达成率 | 7日基线 | 偏离度 |
|------|-------|-------|-------|--------|-------|
| GMV | $75,000 | $95,000 | 78.9% | $105,000 | -28.6% |
| 日环比 | -32.4% | - | - | - | 异常 |

### 影响拆解
**Top 5 下降渠道**:
1. 亚马逊US: -$18,000 (-45%)
2. 独立站: -$8,000 (-28%)
3. 亚马逊UK: -$3,000 (-15%)

**Top 10 影响 SKU**:
1. SKU-001: -$5,000
2. SKU-002: -$3,200
...

### 建议动作
1. ✅ 检查活动日历 → 非活动期
2. ⏳ 检查投放变更 → 待排查
3. ⏳ 检查库存/物流 → 待排查
4. ⏳ 检查退款样本 → 待排查

### 一键操作
- [ ] 生成工单
- [ ] 拉取样本
- [ ] 暂停促销/投放（需权限）
- [ ] 标注状态: 已确认/忽略/待观察
```

### 3. 活动日历与白名单

#### 3.1 活动日历维护

| 项目 | 说明 |
|------|------|
| **维护主体** | 运营/营销团队 |
| **更新频率** | 每周更新，大促前3天完成确认 |
| **对接方式** | 通过共享日历/项目管理工具同步至BI系统 |
| **内容要求** | 活动名称、时间范围、涉及渠道/国家、预期影响 |

**活动日历示例**:
```yaml
activities:
  - name: "Black Friday 2024"
    type: "大促活动"
    start_date: "2024-11-24"
    end_date: "2024-11-28"
    channels: ["亚马逊US", "亚马逊UK", "独立站"]
    expected_impact:
      gmv_increase: "+50%"
      discount_rate: "25-30%"

  - name: "Prime Day 2024"
    type: "大促活动"
    start_date: "2024-07-16"
    end_date: "2024-07-17"
    channels: ["亚马逊全站点"]
    expected_impact:
      gmv_increase: "+40%"
      discount_rate: "20-25%"
```

#### 3.2 活动期阈值调整规则

| 活动类型 | 阈值调整策略 | 示例 |
|---------|------------|------|
| **大促活动** | GMV阈值放宽至±50%，折扣率阈值暂停 | 不因折扣率上升触发告警 |
| **常规促销** | GMV阈值放宽至±40%，折扣率阈值上浮10pp | 折扣率阈值从15pp调整至25pp |
| **品推活动** | GMV阈值放宽至±35%，广告费率阈值上浮20% | 广告费率阈值从30%调整至50% |
| **清仓活动** | 折扣率阈值暂停，毛利率阈值下调5pp | 毛利率阈值从10%下调至5% |

#### 3.3 白名单管理

| 白名单类型 | 适用场景 | 阈值调整 |
|-----------|---------|---------|
| **渠道白名单** | 新渠道上线首月 | 阈值放宽50% |
| **SKU白名单** | 新品上市首2周 | 不纳入告警统计 |
| **国家白名单** | 新开站点首月 | 阈值放宽30% |
| **测试白名单** | A/B测试期间 | 按测试范围调整 |

### 4. SLA响应标准

#### P0 响应SLA

```
响应时限: 1小时内确认告警
根因定位: 4小时内给出根因与初步措施
关闭权限: 业务负责人或其授权方
记录要求: 关闭人、关闭时间、处理结果
```

#### P1 响应SLA

```
响应时限: 4小时内确认
根因定位: 24小时内给出根因与措施
关闭权限: 业务负责人
记录要求: 关闭结果
```

#### P2 响应SLA

```
响应时限: 24小时内确认
处置完成: 72小时内完成处置
关闭权限: 相关部门负责人
记录要求: 关闭结果
```

#### P3 响应SLA

```
响应时限: 1个工作日内确认
处置完成: 3个工作日内完成处置
关闭权限: 业务负责人
记录要求: 简单记录
```

### 5. 告警通知渠道

| 等级 | 通知渠道 | 通知对象 | 频率限制 |
|------|---------|---------|---------|
| P0 | 短信+电话+即时通讯 | 业务负责人+管理层 | 无限制 |
| P1 | 邮件+即时通讯 | 业务负责人 | 同类告警30分钟内合并 |
| P2 | 即时通讯 | 相关部门 | 同类告警1小时内合并 |
| P3 | 日报汇总 | 相关人员 | 每日汇总 |

## 使用方法

### 配置告警规则

```yaml
alert_rules:
  - metric: "gmv"
    conditions:
      - type: "absolute"
        threshold: 85000
        operator: "<"
        level: "P1"
      - type: "relative"
        baseline: "7d_avg"
        threshold: 0.3
        operator: ">"
        level: "P1"
    actions:
      - "pull_top10_sku"
      - "generate_channel_report"
    notifications:
      - channel: "email"
        recipients: ["ops@example.com"]
      - channel: "slack"
        channel_id: "#alerts-p1"
```

### 告警生命周期

```
1. 触发 → 检测到异常条件
2. 生成 → 创建告警卡片
3. 通知 → 发送到对应渠道
4. 确认 → 责任人确认接收
5. 分析 → 定位根因
6. 处置 → 执行改进措施
7. 验证 → 确认问题解决
8. 关闭 → 记录并关闭告警
```

## 示例场景

### 示例1: P0告警处理

**场景**: 毛利额为负触发P0告警

**告警卡片**:
```markdown
## [P0] 毛利额为负 - 紧急告警

### 关键数值
- 毛利额: -$5,000
- 毛利率: -2.1%
- 涉及SKU: SKU-001, SKU-002

### 根因分析
1. SKU-001促销折扣叠加（30%+20%）
2. 采购成本上涨15%未同步调价

### 处置措施
1. ✅ 立即暂停相关促销
2. ✅ 调整SKU-001售价
3. ✅ 修复折扣叠加规则
4. ✅ 通知供应商协商成本

### 验证结果
- 2小时后毛利率恢复至35%
- 告警已关闭
```

## 注意事项

### 告警设计原则

1. **可操作**: 每条告警必须有明确的建议动作
2. **不重复**: 同类告警合并，避免告警风暴
3. **可追溯**: 记录完整的告警生命周期
4. **活动感知**: 活动期自动调整阈值

### 常见问题

**Q: 告警太多怎么办？**
A: 优化阈值、合并同类告警、使用白名单

**Q: 误报率高怎么办？**
A: 增加联动条件、调整阈值、完善活动日历

**Q: 响应不及时怎么办？**
A: 优化通知渠道、升级机制、SLA考核
