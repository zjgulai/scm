# P2 分页与页面长度治理

## 背景

本轮处理用户反馈的两类问题：

- 多个工作台页面纵向长度过长，管理台账型内容缺少分页和序号。
- AI 知识库卡片密度和排版不稳定，卡片窄、高、文本拥挤，影响管理工作台的可读性。

本次只调整前端呈现和 Browser Harness 验收，不改变 SQLite 账本结构，不引入导入能力，不调用外部模型，不写回业务系统。

## 主要改动

- 新增通用 `usePagination` 和 `PaginationBar`，统一支持首页、上一页、下一页、末页和当前区间显示。
- `DataTable` 默认增加序号列，并支持 `rowOffset`，跨页时序号保持全局连续。
- 总览页工作台模块地图分页，治理任务板分页，批量选择改为“选择本页”。
- 工作流编排台的模块契约矩阵、交付关系清单分页，并保留创建编排复核、进入工作台、导出 JSON 等既有动作。
- AI 知识库页的知识源、异常发现、映射矩阵、知识规则、知识卡片全部接入分页；知识卡单页收敛为 6 张。
- 角色工作台的批量治理对象池分页，内部角色队列、Provider、平台就绪度等长清单收纳为卡片内滚动区域。
- 审计日志时间线由 220 条直出改为 10 条/页，payload 预览高度收敛。
- Browser Harness 增加每个页面的 `documentHeight`、`viewportHeight`、`heightRatio`、分页条、序号列、可见卡片数等布局指标。

## 本地验证数据

本地 `SCM_SKIP_PUBLIC_BROWSER_SMOKE=1 npm run smoke:p0` 通过。核心高度诊断结果：

| 页面 | 改造后高度比 | 说明 |
| --- | ---: | --- |
| 治理链路总览 | 9.36 | 模块地图和治理任务板分页，仍保留管理驾驶舱、AI 搜索、任务中心与成熟度扫描 |
| AI 知识库 | 7.35 | 知识卡单页 6 张，5 处分页，知识卡序号和表格序号通过验收 |
| 工作流编排台 | 8.04 | 模块契约 4 条/页，交付关系 4 条/页，编排能力不删减 |
| 角色工作台 | 8.60 | 批量对象池 4 条/页，长清单转为卡片内滚动 |
| 审计日志工作台 | 5.25 | 审计事件 10 条/页，payload 预览收敛 |

历史诊断中的高风险页面已经明显下降：

- 审计日志：37.64 -> 5.25
- 角色工作台：22.64 -> 8.60
- 工作流编排台：11.97 -> 8.04
- 治理链路总览：13.64 -> 9.36
- AI 知识库：9.31 -> 7.35

## 验收边界

- 事实：本地类型检查、生产构建、核心 workflow smoke、Browser Harness smoke 均已通过。
- 事实：Browser Harness 已覆盖 15 个导航页面、4 个响应式视口、无横向溢出、知识库分页布局硬验收。
- 推断：当前页面长度已经从“无限台账堆叠”转为“分页台账 + 卡片内滚动”，后续可以继续把高密度工作台拆为二级页或抽屉式详情。
- 不确定项：公开站点的 SQLite 账本可能比本地 smoke 数据包含更多历史审计事件；审计时间线已分页，理论上不会继续线性拉长页面，但仍需公网 Browser Harness 验收确认。
