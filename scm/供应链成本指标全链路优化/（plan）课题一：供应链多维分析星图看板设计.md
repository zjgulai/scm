## 本地重建说明

本节为基于当前项目本地资料重建的星图看板设计，不等同于钉钉原文复制。它承接星图主题设计和视图类型清单，用于定义 BI 或数据产品页面结构。

## 1. 看板目标

供应链多维分析星图看板要同时服务三类用户：

| 用户 | 主要问题 | 看板入口 |
|---|---|---|
| 管理层 | 成本改善是否真实、哪些事项需拍板 | 经营结果总览 |
| 业务 Owner | 哪个节点异常、该做什么 | 节点治理页 |
| 数据/IT | 口径是否一致、数据是否可用 | 数据与模型页 |

## 2. 页面架构

```text
P0 经营结果总览
P1 成本结构归因
P2 库存健康与现金流
P3 预测计划协同
P4 履约成本与体验
P5 逆向与治理闭环
P6 数据口径与模型监控
```

## 3. 页面设计

| 页面 | 核心模块 | 主要图表 | 必要下钻 |
|---|---|---|---|
| P0 经营结果总览 | 北极星、目标区间、成本对冲、区域差异、待拍板事项 | KPI 卡、趋势线、瀑布、RAG 矩阵 | 区域/渠道/GTM |
| P1 成本结构归因 | 六类成本、四项重算、渠道指纹、负责人矩阵 | 堆叠条、瀑布、热力矩阵、排序表 | 节点/渠道/owner |
| P2 库存健康与现金流 | 在库/在途/未交 PO、库龄、周转、现货满足 | 库龄瀑布、热力图、供需矩阵 | 仓/SKU/区域 |
| P3 预测计划协同 | 预测偏差、PO 偏差、计划达成、供需缺口 | 预测实际趋势、箱线图、PO 漏斗 | 国家/渠道/SKU |
| P4 履约成本与体验 | 头程、尾程、承运商、时效、事故 | 成本时效矩阵、承运商象限、地图 | 国家/渠道/物流商 |
| P5 逆向与治理闭环 | 退货、补发、返仓、责任归因、闭环率 | 帕累托、归因矩阵、闭环漏斗 | 原因/SKU/责任方 |
| P6 数据口径与模型监控 | 指标口径、宽表质量、模型效果、预警命中 | 状态表、漂移图、预警漏斗 | 表/指标/模型 |

## 4. 交互规则

1. 全局筛选：时间、区域、国家、渠道、GTM、SKU、仓、Owner。
2. 所有页面保留同一组核心 KPI：成本率、销售额、成本金额、周转天数、履约满意度。
3. 点击异常点后进入“异常解释面板”，展示第一驱动项、第二驱动项、责任 Owner 和建议动作。
4. 点击节点主题后进入对应八节点治理卡，展示指标、动作、状态、验收。
5. 管理层默认只显示 P0/P1/P5，业务 Owner 默认进入对应节点页。

## 5. RAG 健康规则

| 状态 | 判定 | 页面行为 |
|---|---|---|
| Green | 指标达标且趋势稳定 | 正常展示 |
| Amber | 指标未破红线但趋势恶化 | 展示预警和观察周期 |
| Red | 指标破阈值或连续恶化 | 展示异常卡、Owner、截止日 |
| Grey | 数据缺失或口径未确认 | 禁止输出强结论 |

## 6. MVP 范围

首版只做能支撑经营判断的最小集合：

1. P0 经营结果总览。
2. P1 成本结构归因。
3. P2 库存健康与现金流。
4. P5 逆向与治理闭环的异常动作表。
5. P6 指标口径和数据质量状态表。

P3/P4 可在 Day 60 后随预测、履约宽表成熟后上线。

## 7. `SCM-BI-001` P0 经营结果总览 PRD

### 7.1 页面目标

P0 是管理层入口，不展示所有细节，只回答四个问题：

| 问题 | 页面回答方式 | 不接受的设计 |
|---|---|---|
| 成本改善是否真实 | 北极星指标、同比降本率、目标偏差 | 只放成本率单卡片 |
| 改善是否被对冲 | 成本对冲瀑布展示采购、头程、仓储、尾程变化 | 只展示总成本，不拆节点 |
| 风险在哪里 | 区域/渠道/GTM RAG 矩阵 | 无阈值颜色装饰 |
| 需要谁行动 | 待拍板事项与 Owner 动作表 | 只有洞察文本，没有责任人 |

### 7.2 页面模块

| 模块 | 组件 | 指标 | 主数据源 | 交互 |
|---|---|---|---|---|
| 全局状态栏 | 时间、区域、渠道、GTM、数据质量状态 | `metric_status`、刷新时间 | `dim_scm_metric`、数据质量表 | Grey 状态禁用强结论 |
| 北极星指标组 | 5 个 KPI 卡 + 目标线 | `SC-L1-001`、`SC-L1-002`、`SC-L1-003`、`SC-IV-L4-002`、`FD-L1-001` | `dwt_supply_chain_cost`、`dwt_inventory_health`、`dwt_fulfillment_stability` | 点击 KPI 跳到对应下钻页 |
| 成本对冲瀑布 | YoY 变化瀑布 | 采购、头程、仓储、尾程、逆向、直邮成本变化 | `dwt_supply_chain_cost` | 点击节点跳到 P1 |
| 经营风险矩阵 | 区域 × 渠道 × GTM 热力矩阵 | 成本率、销售额占比、成本增速/销售增速 | `dwt_supply_chain_cost` | 点击单元格打开异常解释面板 |
| 库存与履约护栏 | 小型趋势 + RAG 状态 | 周转天数、缺货率、履约满意度 | `dwt_inventory_health`、`dwt_fulfillment_stability` | Amber/Red 跳到 P2/P4 |
| 待拍板事项 | 状态表 | action、Owner、截止日、预期收益 | `scm_action_tracking` | 只显示 Red 和管理层需拍板动作 |

### 7.3 默认筛选与权限

| 项 | 默认规则 |
|---|---|
| 时间 | 默认最近 MAT 周期，可切换月度 |
| 区域 | 默认全区域，管理层可下钻到区域 |
| 渠道 | 默认全渠道，可切 Amazon、DTC、Offline |
| GTM | 默认全 GTM，可按业务线筛选 |
| SKU | 首页默认不展示 SKU 级明细，只在下钻面板出现 |
| 用户 | 管理层默认进入 P0；节点 Owner 点击异常后进入对应责任页 |

### 7.4 异常解释面板

| 字段 | 说明 |
|---|---|
| `metric_code` | 异常指标编码 |
| `current_value` | 当前值 |
| `target_value` | 目标值 |
| `delta_vs_baseline` | 与基准期差异 |
| `first_driver` | 第一驱动节点或维度 |
| `second_driver` | 第二驱动节点或维度 |
| `owner_name` | 责任人 |
| `recommended_action` | 建议动作 |
| `data_quality_status` | Green/Amber/Red/Grey |

### 7.5 P0 验收

1. 首页必须在一屏内展示“改善、对冲、风险、动作”。
2. `SC-L1-001`、`SC-L1-002`、`SC-L1-003` 必须能从 `dwt_supply_chain_cost` 计算。
3. 任一 Red 指标必须能打开异常解释面板。
4. Grey 指标不得输出强结论和行动建议。
5. 管理层待拍板事项必须绑定 `owner_name`、`due_date` 和验收指标。

## 8. `SCM-BI-002` P1 成本结构归因 PRD

### 8.1 页面目标

P1 是 P0 的成本下钻页，目标是把“成本率异常”拆成可执行归因：

```text
成本异常
  -> 成本节点
  -> 区域 / 渠道 / GTM / SKU 组合
  -> 第一驱动项 / 第二驱动项
  -> Owner / 动作
```

### 8.2 页面模块

| 模块 | 组件 | 视图来源 | 指标 | 主数据源 |
|---|---|---|---|---|
| 六类成本结构 | 堆叠条 + 趋势线 | V02 | 采购、头程、仓储、尾程、逆向、直邮成本率 | `dwt_supply_chain_cost` |
| 四项重算对齐 | 四项费用率趋势 | V02 | `four_item_cost_rate`、`cost_rate` | `dwt_supply_chain_cost` |
| 区域差异瀑布 | 区域贡献瀑布 | V03 | 区域成本率、区域销售额占比 | `dwt_supply_chain_cost` |
| 渠道成本指纹 | 渠道 × 成本节点热力矩阵 | V04 | 节点成本率、节点贡献度 | `dwt_supply_chain_cost` |
| GTM 增长质量 | 增速 × 成本率变动象限 | V05 | 销售增速、成本增速、成本率变动 | `dwt_supply_chain_cost` |
| 负责人责任矩阵 | Owner × 异常组合表 | V06 | 第一驱动、第二驱动、预期收益 | `dwt_supply_chain_cost`、`scm_action_tracking` |
| 国家月份异常 | z-score 趋势 + 异常表 | V07 | 成本率 z-score、销售环比、成本环比 | `dwt_supply_chain_cost` |

### 8.3 归因规则

| 规则 | 说明 |
|---|---|
| 第一驱动项 | 对成本率变化贡献最大的成本节点或维度组合 |
| 第二驱动项 | 排名第二且贡献超过阈值的成本节点或维度组合 |
| 贡献度 | 优先用金额变化和成本率变化同时判断，不只看百分比 |
| 对冲判断 | 如果采购成本上行抵消物流下降，必须在瀑布中显式展示 |
| 异常阈值 | 连续两期恶化或超过目标区间进入 Amber/Red |
| Owner 归属 | 节点 Owner 优先于区域 Owner；跨节点问题进入 PMO 协同 |

### 8.4 下钻路径

| 起点 | 下钻 1 | 下钻 2 | 输出 |
|---|---|---|---|
| P0 成本对冲瀑布 | P1 六类成本结构 | 成本节点明细 | 节点驱动排序 |
| P0 风险矩阵 | P1 渠道成本指纹 | 区域/渠道/GTM 组合 | 高成本组合清单 |
| P1 区域瀑布 | 国家月份异常 | SKU 或成本节点 | 异常月份与原因 |
| P1 负责人矩阵 | 动作台账 | 任务状态 | Owner、动作、截止日 |

### 8.5 P1 验收

1. 每个高成本组合必须显示第一驱动项和第二驱动项。
2. 采购、头程、仓储、尾程四项成本必须能与四项重算口径对齐。
3. 区域、渠道、GTM、SKU 四类维度至少支持三类下钻。
4. 负责人责任矩阵必须能写入或关联 `scm_action_tracking`。
5. z-score 异常只作为线索，不得替代业务归因结论。

## 9. `SCM-BI-003` P2 库存健康与现金流 PRD

### 9.1 页面目标

P2 是库存治理入口，目标是把“库存拖累现金流”拆成可执行的计划、仓储和调拨动作：

```text
库存风险
  -> 在库 / 在途 / 未交 PO
  -> 周转 / 库龄 / 缺货
  -> 仓 / SKU / 区域 / GTM
  -> 调拨 / 清仓 / 补货 / 暂停采购
```

### 9.2 页面模块

| 模块 | 组件 | 指标 | 主数据源 | 交互 |
|---|---|---|---|---|
| 库存现金流总览 | KPI 卡 + 趋势线 | 库存金额、库存周转天数、资金占用成本率 | `dwt_inventory_health` | 点击库存金额下钻到仓/SKU |
| 库龄结构瀑布 | 0-30/31-60/61-90/91-180/180+ 库龄瀑布 | `SC-IV-L3-004`、库龄结构健康度 | `dwt_inventory_health` | 点击 180+ 打开超龄 SKU 清单 |
| 供需缺口矩阵 | 区域 × SKU 或仓 × SKU 热力矩阵 | 现货满足率、缺货率、可售库存 | `dwt_inventory_health` | 点击缺口单元格打开补货/调拨候选 |
| 在途与未交 PO 漏斗 | PO、生产、发运、到港、入仓漏斗 | 未交 PO、在途库存、计划延误 | `dwt_inventory_health`、采购/计划数据 | 点击阶段定位阻塞责任方 |
| 调拨候选池 | 排序表 | 调拨收益、缺货风险、超龄风险、调拨成本 | `dwt_inventory_health`、`scm_action_tracking` | 生成候选动作，不自动执行 |
| 库存动作台账 | 状态表 | action、Owner、截止日、预期释放金额 | `scm_action_tracking` | Red 风险必须绑定动作 |

### 9.3 库存风险规则

| 风险 | 判定 | 页面动作 |
|---|---|---|
| 超龄风险 | 90+ 或 180+ 库存金额超过目标区间 | 进入清仓、调拨或降价动作池 |
| 缺货风险 | 缺货率连续恶化或现货满足率低于目标 | 进入补货或跨仓调拨候选 |
| 资金占用风险 | 周转天数上升且销售增速放缓 | 标记采购/计划复盘 |
| 错配风险 | 某区域缺货，同时其他区域超龄 | 优先生成调拨候选 |
| 数据风险 | 在库、在途、未交 PO 任一口径缺失 | 显示 Grey，不输出动作建议 |

### 9.4 下钻路径

| 起点 | 下钻 1 | 下钻 2 | 输出 |
|---|---|---|---|
| P0 库存护栏 | P2 库存现金流总览 | 区域/仓/SKU | 库存风险排序 |
| P2 库龄瀑布 | 超龄 SKU 清单 | Owner/渠道/GTM | 清仓或调拨动作 |
| P2 供需缺口矩阵 | 缺货 SKU 清单 | 在途/未交 PO | 补货或调拨候选 |
| P2 调拨候选池 | 动作台账 | 任务状态 | Owner、收益、截止日 |

### 9.5 P2 验收

1. `SC-IV-L4-002`、`SC-IV-L3-004`、`FD-IV-L4-003` 必须能从 `dwt_inventory_health` 计算。
2. 90+/180+ 库存必须能下钻到仓、SKU、区域和 Owner。
3. 缺货风险不得只看库存数量，必须联动可售库存、在途和未交 PO。
4. 调拨候选必须展示收益、成本、风险和人工确认状态。
5. Grey 数据状态不得生成清仓、补货或调拨建议。

## 10. `SCM-BI-004` P5 逆向与治理闭环 PRD

### 10.1 页面目标

P5 是逆向治理和整改闭环入口，目标是把“退货补发是否重复发生”拆成原因、责任、价值恢复和动作状态：

```text
逆向成本
  -> 退货 / 补发 / 报废 / 返仓
  -> 原因编码 / 责任归属
  -> SKU / 渠道 / 国家 / 批次
  -> 整改动作 / 返仓恢复 / 闭环验收
```

### 10.2 页面模块

| 模块 | 组件 | 指标 | 主数据源 | 交互 |
|---|---|---|---|---|
| 逆向成本总览 | KPI 卡 + 趋势线 | 退货处理成本率、补发成本率、返仓可售率 | `dwt_reverse_logistics` | 点击成本项进入原因分解 |
| 退货原因帕累托 | 帕累托图 | 退货率、退货金额、退货处理成本 | `dwt_reverse_logistics` | 点击原因进入 SKU/渠道明细 |
| 责任归因矩阵 | 原因 × 责任方热力矩阵 | 责任占比、重复发生率 | `dwt_reverse_logistics` | 点击单元格打开整改动作 |
| 返仓价值漏斗 | 退货申请、收货、质检、可售、再销售漏斗 | 返仓可售率、处理时效、报废率 | `dwt_reverse_logistics`、WMS | 识别价值损失阶段 |
| TOP SKU 逆向清单 | 排序表 | 退货率、补发率、损失金额、责任方 | `dwt_reverse_logistics` | 进入 SKU 整改任务 |
| 闭环动作台账 | 状态表 + 漏斗 | action、Owner、截止日、闭环率 | `scm_action_tracking` | TOP 原因必须绑定整改状态 |

### 10.3 闭环规则

| 状态 | 判定 | 页面动作 |
|---|---|---|
| New | 新增 Red 逆向异常 | 自动生成待确认动作草稿 |
| Assigned | 已确认 Owner 和截止日 | 进入动作台账 |
| In Progress | Owner 已提交整改动作 | 展示阶段进度 |
| Verified | 指标回落或问题不再重复出现 | 标记验收完成 |
| Reopened | 同一原因再次 Red | 重新打开并升级到 PMO |

### 10.4 下钻路径

| 起点 | 下钻 1 | 下钻 2 | 输出 |
|---|---|---|---|
| P0 待拍板事项 | P5 闭环动作台账 | TOP 原因 | 责任人和整改状态 |
| P5 退货原因帕累托 | SKU/渠道/国家明细 | 责任归因矩阵 | 根因和责任方 |
| P5 返仓价值漏斗 | 阶段损失明细 | 仓/质检/客服动作 | 价值恢复动作 |
| P5 TOP SKU 清单 | SKU 逆向画像 | 动作台账 | 整改动作和验收指标 |

### 10.5 P5 验收

1. 退货、补发、报废、返仓可售必须能拆到原因编码。
2. TOP 原因必须绑定责任方、Owner、截止日和验收指标。
3. 返仓价值漏斗必须区分可售、不可售、待质检和报废。
4. 重复发生的 Red 异常必须重新打开，并升级到 PMO 协同。
5. 无原因编码的退货不得进入强结论，只能进入数据治理清单。

## 11. 验收标准

1. 管理层能在首页看到“改善、对冲、风险、动作”四类信息。
2. 每个红色异常必须能下钻到节点、指标、Owner、动作。
3. 看板引用的指标必须在指标字典中存在。
4. 数据缺失必须显示 Grey，不允许用空值生成误导结论。
5. 看板输出必须能直接支撑 `供应链成本效率整合洞察及方案V1.0`。
