## 任务 0.1 · 安全加固：排查并保护 .env

**债务**：V2 — .env 文件存在，内含 API Key，虽在 .gitignore 但存在误提交风险；SECURITY.md 中 DOUBAO_APP_KEY 等引用暗示可能存在更多密钥。

**步骤**：

1. **确认 .env 是否已被 git 追踪**
   ```bash
   git ls-files .env
   ```
   - 若有输出 → 执行 `git rm --cached .env`，然后检查 git log 确认密钥是否已入历史
   - 若已入历史 → 去 Biji 后台轮换 API Key，在 .env.example 中更新占位符

2. **审计密钥分布**
   ```bash
   grep -rn "API_KEY\|TOKEN\|SECRET\|PASSWORD\|ACCESS_KEY" --include="*.py" --include="*.md" --include="*.sh" | grep -v '.git/' | grep -v 'node_modules' | grep -v 'example'
   ```
   对每个命中判断是否为真实密钥，若是则立即轮换并从仓库移除。

3. **加固 .gitignore**
   确认以下条目存在且生效：
   ```
   .env
   *.pem
   *.key
   credentials.json
   service-account.json
   ```
   添加 `ref/aippt/` 和 `ref/books/` 到 .gitignore 或评估是否用 Git LFS。

4. **验证**
   ```bash
   git status --short | grep -E '\.env$|\.pem$|\.key$|credentials|service-account'
   ```
   应无输出。

**产出文件**：`.gitignore`（修改）、`.env`（确认安全）

2. **创建 `configs/path_loader.py`**（供 Python 脚本使用）
   ```python
   """路径加载器 — 从 paths.yaml 读取所有项目路径"""
   from pathlib import Path
   import yaml

   _CONFIG = None

   def load_paths():
       global _CONFIG
       if _CONFIG is None:
           config_file = Path(__file__).resolve().parent / "paths.yaml"
           with open(config_file) as f:
               _CONFIG = yaml.safe_load(f)
       return _CONFIG

   def get_path(key: str) -> Path:
       """通过点号路径获取，如 'data_sources.analysis_master'"""
       config = load_paths()
       parts = key.split(".")
       node = config
       for part in parts:
           node = node[part]
       repo_root = Path(__file__).resolve().parents[1]
       return repo_root / node
   ```

## 任务 2.1 · 架构决策：三层框架的存废

**债务**：E1 — Agent/Engine/Output 是纸面框架，实际分析链路绕过它。

**决策流程**：

这是一个关键的架构决策点，需要在动手前明确。以下是决策框架：

### 选项 A：保留并完善三层（推荐）

**前提**：三层框架的设计意图是正确的——意图解析 → Skill 路由 → 数据处理 → 结果输出——是 Data Agent SaaS 的核心价值。

**投入**：任务 2.2-2.4 的工作量。将 `data_processor.py` 从 Mock 数据切换到真实数据读取和计算，让 PPT 生成真正输出 .pptx。

**收益**：
- 获得统一的 CLI 入口：`python -m main_project_lute.agent "分析 DTC 毛利率"`
- 未来新增分析能力只需注册 Skill，不需要写全新脚本
- 与 "AI SaaS" 产品定位一致

**风险**：工作量大，可能超过阶段 2 的预算

### 选项 B：废弃三层，保留直接脚本执行模式

**前提**：当前的 `data_example/scripts/` 直接执行模式更简单、更可靠。三层框架是过度设计。

**投入**：删除 `agent/`、`engine/`、`output/` 目录，清理 CLAUDE.md 中相关引用，将 `skills_router.py` 的有用部分（关键词映射表）提取为独立工具。

**收益**：代码量减少，维护负担降低

**风险**：失去产品化路径，AI SaaS 概念退化为脚本集合

### 选项 C：保留三层作为 API 层，底层委托给现有脚本

**折衷方案**：保留 `agent/` 作为入口（意图解析 + Skill 路由），但 `engine/data_processor.py` 改为调用现有的 `data_example/scripts/` 脚本（subprocess），`output/ppt_generator.py` 改为读取已生成的 .pptx 并组合。三层变成编排层而非重新实现层。

**推荐：选项 C**——既保留了 AI SaaS 的产品架构，又避免了重写所有分析逻辑。实施成本最低。

**产出**：架构决策记录（ADR）写入 `docs/architecture/decisions/001-agent-architecture.md`

   ```python
   def _process_margin_attribution(self, params: Dict) -> Dict:
       """毛利率归因分析 → 委托给专题01脚本"""
       from path_loader import get_path
       script = get_path("main_project.data_example") / "scripts/core/run_专题01_平台区域毛利归因.py"
       output_dir = get_path("outputs.tmp") / "margin_attribution"

       result = subprocess.run(
           [sys.executable, str(script),
            "--platform", params.get("platform", "DTC"),
   ```
