# 四大专题完整治理与产品化执行计划草稿

## 1. 已确认决策

| 决策项 | 结论 | 执行含义 |
|---|---|---|
| D1 供应链定位 | `scm/` 正式定义为专题②的独立深挖分枝，保持顶层专题库，不迁入 `main_project_lute/` | 专题②蓝图需要引用 SCM，但不吞并 SCM |
| D2 推进顺序 | 同意先做专题②，但必须保存四大专题完整 plan | 本文件作为四大专题主线锚点，后续单专题推进不得偏离 |
| D3 样例产物映射 | `data_example/专题产物/专题01-05` 做映射表，不提升为主专题 | 样例产物只作为证据、方法和脚本复用来源 |
| D4 蓝图落位 | 先在 `drafts/analysis/` 完成控制面板，经确认后再转正式目录 | 当前不新增正式专题目录 |
| D5 命名前缀 | 采用 `VOC-*`、`ORDER-*`、`CHANNEL-*`、`MKT-*`、`XL1-*` 到 `XL4-*`、`SCM-*` | 每个专题独立治理，不交叉污染 |

## 2. 项目主线

当前项目主线固定为：

```text
数智决策-AI效能作战室
  -> 4 大业务专题
  -> 14 个子课题
  -> 4 条交叉故事线
  -> 5+1 智能体
  -> 全局数据契约
  -> Phase2 / Phase3 管道
  -> 运营化与会议消费
```

本计划服务于 `main_project_lute/` 的四大专题，不替代 `scm/` 已完成的供应链专项，也不把早期样例专题产物误判为新的业务主专题。

## 3. 证据边界

| 证据层 | 当前状态 | 允许结论 | 禁止结论 |
|---|---|---|---|
| Phase1 故事线 | 4 个专题故事线、4 条交叉故事线已存在 | 可作为业务需求与验收来源 | 不代表数据真实可用 |
| 全局数据契约 | 01 数据需求矩阵、05 数仓主键设计已存在 | 可作为表字段规划基线 | 不代表生产库表已确认 |
| Phase2 / Phase3 状态 | 文档显示专题跑通，并有 mock/output | 可证明 mock 或阶段产物链路存在 | 不代表生产数仓接入完成 |
| SCM 深挖分枝 | SCM 已完成 Source / DQ / SQL 前置规格 | 可作为治理模板 | 不代表四大专题均已完成同等治理 |
| `data_example/专题产物` | 早期专题产物和脚本较多 | 可复用方法、图表、样例证据 | 不可提升为宏观业务主专题 |

## 4. 总体目标

把四大专题从“规划文档 + mock 产物”推进为可治理、可执行、可验证的产品化链路。

完成后，每个主专题都必须具备：

```text
分层蓝图
  -> 子课题清单
  -> 指标字典
  -> 主题宽表规格
  -> 看板 / 分析视图 PRD
  -> Agent 任务规格
  -> Runtime 输出边界
  -> 真实源表确认矩阵
  -> 真实源系统确认包
  -> DQ 规格
  -> SQL 前置规格
```

交叉线只在上游专题产出稳定后治理，不重建上游专题，不越权输出上游结论。
