# 供应链履约时效看板 PRD 草稿

## 1. 产品目标

建立一个以“指标体系与指标字典”为前置的供应链履约时效看板，覆盖 Excel sheet 2 的业务需求，并与本地 SCM 知识库的 MECE 指标体系对齐。

本轮目标是本地原型，不接生产库、不写回外部系统、不调用 provider。

## 2. 用户与场景

| 用户 | 主要问题 | 看板价值 |
|---|---|---|
| 供应链负责人 | 今日履约是否失控，风险在哪里 | 快速看总览、趋势、异常和责任域 |
| 订单履约/仓库负责人 | 哪些订单卡住，卡在哪个环节 | 从 KPI 下钻到订单明细 |
| 物流负责人 | 哪些国家、仓库、渠道妥投慢 | 看区域渠道妥投和跨区发货 |
| 计划/商品负责人 | 哪些 SKU 长期慢发或缺货 | 看商品履约分布和缺货分析 |
| 数据/BI Owner | 指标口径、来源表和字段是否明确 | 用指标字典和数据表清单落 SQL |

## 3. 信息架构

默认首屏为 `指标体系`，而不是业务故事线。顺序如下：

1. 指标体系与指标字典
2. 履约总览
3. 订单履约明细
4. 订单履约分布
5. 区域渠道妥投分析
6. 商品履约分布
7. 发货效能
8. 审核效能
9. 异常订单-问题件
10. 未发货预警
11. 缺货分析
12. 跨区发货
13. 故事线、PRD、数据表

## 4. 筛选器

| 筛选项 | 交互 | 说明 |
|---|---|---|
| 付款时间 | 时间区间 | 时间按照北京时间，刷新频率按天 |
| SKU | 模糊搜索 | 需要支持 code/name |
| 产品名称 | 模糊搜索 | 关联 SKU 维表 |
| 店铺 | 多选 | 线上店铺/账号 |
| 站点 | 多选 | 平台站点 |
| 国家/区域 | 多选 | 国家维表关联区域 |
| 仓库 | 多选 | 发货仓库 |
| 物流方式 | 多选 | shipping code/name |
| 订单状态 | 多选 | 大状态 + 小状态 |
| 异常标识 | 多选 | order_status 小状态 |
| 预售状态 | 多选 | 订单标签 |
| 末次审单人 | 多选 | 待审核日志完善 |
| 是否虚拟商品自动发货 | 是/否 | 虚拟商品不剔除但需标记 |

## 5. 业务主题故事线

Excel sheet 2 的十大主题是主线；按最新业务决策，`未发货预警` 额外保留为独立模块，不再合并进缺货或明细。

| 主题 | 先看什么 | 要解决的问题 | 核心图表 |
|---|---|---|---|
| 指标速览 | 订单规模、及时率、时长、异常 | 今天履约是否失控 | 横向 KPI 卡片、趋势组合图、诊断卡 |
| 订单履约明细 | 平台单号、系统单号、状态、SKU、时点 | 每一单卡在哪个环节 | 高密度明细表、口径抽屉 |
| 订单履约分布 | 付款日期、区域/国家、SKU、订单状态 | 哪些结构导致发货率差异 | 热力图、排名表、分布表 |
| 区域渠道妥投分析 | 包裹数、签收数、3/5/7/10/12 天签收率 | 哪些区域/渠道妥投慢 | 妥投阶梯、渠道矩阵 |
| 商品履约分布 | SKU、SPU、品类、GTM 组、商品维度发货率 | 哪些商品长期慢发 | SKU 表、慢发排行 |
| 发货效能 | 发货单量、商品数量、小包仓、海外仓 | 仓型是否承压 | 仓型堆叠条、日趋势 |
| 审核效能 | 审核方式、审核人、累计审核单量、平均审核间隔 | 未审核是否由系统自动审核、人工人效或系统日志导致 | 系统/人工对比、审核人排行、时长分布 |
| 异常订单-问题件 | 异常标识、问题件类型、数量、比例 | 异常要怎么闭环 | 问题件帕累托、明细下钻 |
| 未发货预警 | 超期未发货订单、7/10/15 天分层、仓库、SKU、Owner | 哪些已付款订单超过 SLA 仍未发货 | 超期分层表、责任域卡片、明细下钻 |
| 缺货分析 | 订单缺货、库存缺货、预测缺货 | 缺货来自履约异常、当前可售不足还是未来覆盖不足 | 三类缺货卡、SKU 风险表、库存覆盖趋势 |
| 跨区发货 | 目的国家、发货仓国家、跨区率 | 跨区对时效和成本的影响 | 国家-仓库矩阵、跨区排行 |

## 6. 图表设计

| 图表 | 用途 | 设计规则 |
|---|---|---|
| 横向 KPI 卡片 | 展示核心指标，保持 Readdy 原型风格 | 白底、灰线、8px 圆角；比率指标展示分子分母 |
| 趋势组合图 | 对比单量和及时率 | 单量用绿色实心柱，及时率用红色点/虚线 |
| 区域热力 | 订单履约分布 | 绿色深浅表达订单占比 |
| 排名表 | 国家/区域、问题件、SKU 排名 | 数量 + 占比 + 横向条 |
| 妥投阶梯 | 3/5/7/10/12 天签收 | 进度条显示累计达成 |
| 仓型堆叠条 | 小包仓与海外仓处理量 | 绿色/蓝色分段 |
| 跨区矩阵 | 目的地与发货仓错配 | 高风险红底、正常绿底 |
| 未发货预警表 | 7/10/15 天未发货订单治理 | 保留订单数、占比、优先动作和等级 |
| 三类缺货卡 | 订单缺货、库存缺货、预测缺货 | 不把三类缺货混成一个字段 |

### Phase 1：口径确认

- 缺货口径已确认：订单缺货、库存缺货、预测缺货三类。
- 签收时间权威源已确认：TMS。
- 平均发货成本已确认：V0 占位，不作为核心指标。
- 未发货预警已确认：保留独立模块。
- 审核效能已确认：必须区分系统自动审核与人工审核。
- 数据 owner 仍需确认上述口径对应的真实字段、同步频率、数据质量和权限。

### Phase 3：看板工程化

- 将静态原型迁移到正式 React/Vite 或现有 BI 前端。

## 10. 已确认关键口径

| 问题 | 当前决策 |
|---|---|
| 缺货分析是否分成订单缺货、库存缺货、预测缺货三类？ | 是，三类都进入指标体系 |
| 妥投率签收时间来源以 TMS 为准还是物流商后台为准？ | 以 TMS 为准 |
| 平均发货成本本轮是否进入核心 KPI？ | V0 只占位，不作为核心指标 |
| 未发货预警是否永久删除独立模块？ | 否，保留独立模块 |
| 审核效能是否必须区分系统自动审核与人工审核？ | 必须区分 |

## 11. 边界声明

事实：本轮已完成本地静态原型、指标字典、PRD、执行计划，并把 11 个业务模块都推进为页面级洞察图表；同时已补充图表数据绑定矩阵、表级只读契约和 SQL/BI 口径样例。

推断：当前最稳妥的工程路径是先用只读订单事实表跑通总览和明细，再接 TMS、审核事件、库存、预测和责任域，最后做页面级验收。

不确定项：上述业务口径已确认，但 TMS、库存、预测、物流成本、审核日志的真实字段与同步质量仍需数据 owner 验证；未验证前不能声明生产数据口径已完成。
