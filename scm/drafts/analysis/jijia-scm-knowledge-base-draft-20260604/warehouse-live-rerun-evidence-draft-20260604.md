# 积加 ERP 仓库模块二次重采证据摘要

## 结论

本轮按用户纠正重新采集仓库导航下 40 个子页面。上一轮把部分页面误判为无数据，根因是页面加载慢、默认筛选未稳定、以及状态页签和表格列表不一定同步出现。本轮改为等待分页总数、表格行、非零合计窗口或稳定空表信号后再记录。

原始提取文件：`tmp/outputs/jijia-warehouse-live-extraction-20260604.json`。

## 重采规则

- 只读读取页面可见字段、筛选器、页签、按钮和表格，不执行新增、提交、导入、下单、发货、确认等会改变业务状态的动作。
- 页面必须等待到 `共 N 条`、可见表格数据、非零合计窗口、非零状态页签，或稳定 `暂无数据` 后才记录。
- 当前筛选为空不等于业务无历史数据；它只说明当前页面筛选条件下没有返回列表。
- 页面分页总数和状态页签数量不能直接当最终指标口径；它们是 UI 证据，需要后续导出或 API 校验。

## 异常与待校验项

- `入库单` 页签显示 `待入库:818`，但当前表格列表为 0；说明入库单还受质量状态、仓库、时间或其他隐藏筛选影响，不能按列表 0 判断入库业务为空。
- `库龄分析` 在当前自带浏览器会话重置后仍为空；用户本地 Chrome 截图曾显示库龄数据，因此库龄页必须后续单独按正确仓库/平台筛选复采。
- `三方仓销售出库单` 表格有可见记录样本，但分页总数和状态页签均显示 0；该页需要用导出或 API 校验真实分页口径。
- `包裹单` 分页总数为 55937，而页签 `全部` 为 1626152；二者明显不是同一筛选口径，建模时必须拆分为列表筛选口径和状态总览口径。
- `扫码装箱`、`扫码发货`、`平台组包` 是作业入口，不适合按列表页抽取数据字典，应进入操作流程知识库。

## 对指标体系的修正

1. 仓库模块不是无数据页面集合；库存、库内、出库、三方仓、稽核和主数据均存在可采字段和可见业务量。
2. 指标建设优先级应从库存快照、批次库存、调拨/调整/出库、包裹物流、三方仓入库、仓库主数据开始。
3. 当前 UI 证据适合建设页面字典、实体字典、指标字典和血缘草图；最终字段类型、主键、隐藏字段、刷新频率必须通过导出文件或 API 继续确认。
4. 页面状态页签数量不能直接并入分页总数；它们应作为单独的 `status_snapshot` 或 `order_status_count` 口径处理。
