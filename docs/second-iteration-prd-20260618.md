# 供应链数据开发治理工作台二次迭代 PRD

## 1. 决策固化

本轮迭代按用户确认的 5 条边界执行：

- 先不增加登录。
- 先写 SQLite 工作台账本。
- 先做本地检索和证据返回，再接 DeepSeek/Kimi 其中一个。
- 本体只读，只允许注解、评论、修订建议。
- 同意先完成二次迭代规划，再进入代码实现。

事实边界：

- 当前线上站点是原型，不是正式生产治理系统。
- 当前知识库仍位于 `drafts/analysis/`，应作为治理输入和证据来源，不直接视为已验证生产口径。
- 当前指标字典 2.0 以 MECE V2 蓝图为 canonical 基准，本轮不直接改写 canonical 指标，只在台账中记录注解、评论和修订建议。

## 2. 产品定位

二次迭代目标不是再做一个展示型门户，而是把原型推进为“可操作的数据治理工作台”：

- 每个工作台有明确的查询、查看、创建、编辑、评论、修订建议、流程状态和审计记录。
- 核心数据资产以 SQLite ledger 承载，不依赖外部登录和复杂权限体系。
- AI 能围绕本地知识库做检索、证据返回和拒答，不直接自由访问原始表。
- 指标体系用画布表达 L0-L3 结构，支持点击、注解、证据、血缘和修订建议。
- UI 从深色后台风格调整为浅色咨询公司风格，参考 Umbrex 页面气质：留白、细线、低饱和背景、清晰信息层级。

## 3. 用户与场景

| 角色 | 核心目标 | 高频操作 |
|---|---|---|
| 供应链数据负责人 | 建立可信指标体系和治理闭环 | 查看治理健康度、审批修订建议、推动 owner 认领 |
| BI/指标 owner | 维护指标口径、维度、血缘和质量状态 | 查询指标、补充口径、提交字段映射建议、评论异常 |
| 供应链运营分析师 | 用指标和知识库支撑诊断 | 检索知识、提问 AI、查看证据链、发起洞察任务 |
| 数据开发工程师 | 把业务语义落到数据资产 | 查看物理映射、血缘、质量规则、生成开发任务 |
| 业务 owner | 判断口径是否符合业务事实 | 审阅注解、评论规则、确认阈值和口径建议 |

## 4. 范围

### 4.1 本轮包含

- 为所有工作台补齐基础工作流。
- 新增 SQLite 台账表，用于注解、评论、修订建议、流程、AI 问答和知识库检索证据。
- 新增 AI 知识库导航，并按主题域组织三大供应链知识库。
- 新增本地 AI 对话入口，先以检索和证据返回为主。
- 将指标体系改造为可点击画布。
- 重做浅色专业咨询风格 UI。

### 4.2 本轮不包含

- 不增加登录和复杂 RBAC。
- 不直接写回积加、ERP、WMS、TMS 或生产数据库。
- 不直接修改指标字典 2.0 canonical 数据。
- 不自动改写本体对象，只允许注解、评论和修订建议。
- 不接入自由 NL2SQL。
- 不在本轮默认接 DeepSeek/Kimi；接口预留，等本地检索证据链稳定后再接。

## 5. 工作台模块与交互工作流

| 模块 | 主要对象 | 工作流 |
|---|---|---|
| 治理链路总览 | 治理健康度、待办、风险 | 筛选治理域 -> 查看资产状态 -> 创建治理任务 -> 记录处理意见 -> 查看审计轨迹 |
| 对象本体工作台 | SKU、Listing、Supplier、PO、Warehouse 等对象类型 | 查看对象 -> 查看属性/关系/状态 -> 添加注解/评论 -> 提交修订建议 -> 等待人工确认 |
| 标签工程工作台 | 规则标签、统计标签、模型标签 | 新建标签草稿 -> 绑定对象类型 -> 补充规则和证据 -> 提交 review -> 认证/退回/废弃 |
| 维度工程工作台 | 一致性维度、分析维度、层级 | 新建/编辑维度 -> 维护层级和适配指标 -> 增加质量约束 -> 提交认证 |
| 指标工程工作台 | 原子、派生、复合指标候选 | 新建指标候选 -> 绑定公式/粒度/维度 -> 补充字段映射建议 -> 触发质量检查 -> 提交 owner 审阅 |
| 指标字典工作台 | 指标字典 2.0 canonical 指标 | 只读检索 -> 查看口径/owner/维度/血缘 -> 添加注解/评论 -> 提交修订建议 |
| 指标体系画布 | MECE V2 L0-L3 节点 | 画布浏览 -> 点击节点 -> 右侧查看注解、证据、血缘、适用维度 -> 提交修订建议 |
| 血缘与质量工作台 | 字段血缘、指标血缘、DQ 规则 | 查看血缘 -> 标记质量问题 -> 绑定影响指标 -> 生成治理任务 -> 记录处理结果 |
| AI 知识库 | 三大知识库主题域、卡片、来源 | 浏览主题域 -> 查看知识卡 -> 搜索证据 -> 建立指标/对象/规则 crosswalk -> 提交整理建议 |
| AI 对话 | 本地检索问答、证据返回 | 选择知识范围 -> 输入问题 -> 返回答案/证据/无法证明原因 -> 保存为洞察或修订建议 |
| ChatBI 语义治理台 | 可问指标、语义上下文、拒答规则 | 查看可问清单 -> 维护问法样本 -> dry-run 问题 -> 查看拒答原因和证据链 |
| 决策闭环工作台 | 洞察、建议、审批、任务、复盘 | 新建洞察 -> 生成建议 -> 创建行动任务 -> 记录执行反馈 -> 复盘有效性 |

## 6. AI 知识库主题域规划

三大知识库保持物理隔离，但在工作台中通过主题域做统一检索：

| 主题域 | 主要来源 | 业务价值 |
|---|---|---|
| 积加系统语义域 | 积加 SCM 主知识库 | 对齐计划、采购、仓库、物流模块的系统页面、字段、流程和指标来源 |
| 备货库存规则域 | 备货库存规则知识库 | 固化计划库存、采购未交、在途、在库业务库存等业务库存计算规则 |
| 供应链运营方法域 | 业务知识库 | 沉淀跨境电商全链路运营、指标、算法、场景和治理方法 |
| 指标体系蓝图域 | 业务知识库 + BI 字段萃取 + 旧指标体系 | 支撑 L0-L3 指标体系、口径治理和指标 owner 认证 |
| 数据质量与血缘域 | 三大知识库交叉成果 | 支撑字段映射、血缘追踪、异常解释和证据链返回 |
| 决策场景域 | 业务知识库场景卡 + 备货规则 | 支撑补货、库存健康、采购履约、物流履约、成本和异常闭环 |

## 7. 指标体系画布要求

画布不是静态图，而是指标治理入口。

- 左侧：L0/L1 筛选、状态筛选、owner 筛选、认证状态筛选。
- 中间：L0-L3 节点图，默认展示 L0/L1/L2，点击后展开 L3。
- 右侧：节点详情抽屉，展示定义、公式、粒度、方向、可用维度、血缘、知识库证据、注解、评论和修订建议。
- 底部：证据条，展示来源知识库、来源文件、字段映射、质量状态。
- 交互：节点搜索、展开/折叠、定位、注解、评论、修订建议、复制引用。

## 8. UI 风格要求

视觉方向：

- 使用浅色侧边导航，取消深色侧边栏。
- 背景使用微暖灰，不使用大面积纯白或深黑。
- 组件使用细线、低阴影、克制色块。
- 信息密度适中，接近咨询公司资料库和专业 BI 治理台。
- 页面层级清晰：顶部全局控制、左侧导航、中间主工作区、右侧上下文抽屉。

建议设计 token：

| Token | 值 | 用途 |
|---|---:|---|
| `color.background` | `#F7F4EF` | 页面背景 |
| `color.surface` | `#FFFFFF` | 卡片和表格 |
| `color.ink` | `#2F353B` | 正文 |
| `color.navy` | `#203A5F` | 主导航和标题强调 |
| `color.muted` | `#6C7A89` | 次级文本 |
| `color.line` | `#DDD6CC` | 分割线 |
| `color.accent` | `#B65A3C` | 主行动和重点状态 |
| `color.sage` | `#5E7D62` | 已认证、通过 |
| `color.ochre` | `#B8872B` | 风险、待审 |

## 9. 验收标准

- 每个工作台都有可执行工作流入口，不再只是只读展示页。
- SQLite 台账能记录注解、评论、修订建议、流程状态和 AI 检索证据。
- 本体和指标字典 2.0 canonical 资产保持只读。
- AI 对话必须返回证据；证据不足时返回“样本无法证明/证据不足”，不编造结论。
- 指标体系画布支持点击节点、查看详情、提交注解和修订建议。
- AI 知识库能按主题域浏览三大知识库，并保留物理来源路径。
- UI 改为浅色咨询风格，侧边栏不再使用深色底。
- 不产生生产写入、不调用外部模型、不写回业务系统。
