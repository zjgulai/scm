# 001 项目级 Skills 优化全过程复盘（大白话版）

- 日期：2026-03-16
- 主题：把 `.cursor/skills/` 里的项目级 Skills 从“能用”升级到“好用、稳用、少误触发”

## 一句话先说结论

我们把项目里的 6 个 Skills 做了系统升级：
- 触发描述更清楚（什么时候该触发、什么时候不该触发）
- 文件结构更统一（每个 Skill 都有说明、参考、示例、触发测试）
- 旧路径/旧命令都修掉了
- 最后还做了“误触发压力测试”，并加了防误触发边界

通俗讲：**现在这套 Skills 更像一个有标准、可维护、能持续迭代的“工具箱”，不是散装文档。**

---

## 背景问题（当时哪里不舒服）

我们先盘点了 `.cursor/skills/`，发现三类问题：

1. **成熟度不一致**
   - 有的 Skill 比较完整
   - 有的只有 `SKILL.md`，缺少 `reference/examples/trigger-evals`

2. **触发边界不清晰**
   - 有些 description 写法偏“我能干什么”，不够“什么时候该触发”
   - 容易出现相邻 Skill 抢任务（误触发）

3. **内容有过时信息**
   - 尤其 `momcozy-*` 里还残留旧目录路径（`data_example/data`、`docs`、`outputs`）
   - 但项目已经改成新结构（`原始数据/专题产物/...`）

---

## 这次我们到底做了什么（按步骤）

### 第 1 步：先做全量体检（Audit）

- 把 6 个项目级 Skill 全部读了一遍
- 查了每个目录有没有 `reference.md`、`examples.md`、`trigger-evals.json`
- 对照你刚完善的 `ClaudeSkillsCreateSkills` 标准，找出不达标点

### 第 2 步：统一主文件写法（Standardize）

把 6 个 `SKILL.md` 统一成同一套骨架：
- `When to Use`
- `When Not to Use`
- `Core Workflow`
- `Common Mistakes`
- `Resources`

并把 description 统一成 **Use when...** 的触发导向写法。

### 第 3 步：补齐配套文件（Add support files）

给不完整的 Skill 补上：
- `reference.md`（长规则/口径/模板）
- `examples.md`（典型输入输出）
- `trigger-evals.json`（正负触发样例）

结果：6 个 Skill 现在全部都是“四件套”。

### 第 4 步：修复过时路径（Repair stale paths）

重点修了 `momcozy-margin-analysis` 和 `momcozy-product-line-basket-reporting`：
- 路径改到当前真实结构
- 命令改到当前可执行脚本入口
- 清理掉旧目录字面引用

### 第 5 步：重写索引（Refresh README）

更新 `.cursor/skills/README.md`，让它和真实目录一致：
- 不再写错“每个 Skill 都有啥”
- 补了调用顺序、分组、边界

### 第 6 步：做触发压力测试（Verify + Stress）

- 每个 Skill 增补了 near-miss（高混淆）样例
- 生成了压测报告：
  - `/.cursor/skills/TRIGGER_EVAL_STRESS_REPORT.md`
- 又加了一轮反向边界优化：
  - 每个 description 增加 `Not for ...`
  - 并同步到每个 `trigger-evals.json` 的首条边界样例 `[Boundary] ...`

## 最终结果（你最关心的）

### 已达成

- 6/6 Skill 结构统一
- 6/6 description 有清晰触发语义
- 6/6 都有触发评测样例
- `momcozy-*` 不再引用旧路径
- 有了可复用的触发压测报告

### 直接收益

- 误触发概率下降
- 新同事接手成本下降
- 后续改 Skill 时有“标准动作”可照着走

## 给未来自己的“听得懂版本”

如果你之后忘了今天做了什么，只记这 4 句：

1. **先把 Skill 写清“什么时候触发”，别先讲自己多强。**
2. **每个 Skill 都要有四件套，不然迟早失控。**
3. **路径和命令必须跟当前项目真实结构同步。**
4. **改完一定做触发压测，不测就等于没改完。**
