# 项目技术负债 & 脆弱点深度审计报告

> 审计范围：`/Users/pray/project/ecom_ana_overview` 全仓库
> 审计日期：2026-06-09
> 仓库规模：2.8GB（不含.git），~16800 行 Python，677 个 Markdown 文件，158 个 SKILL.md，39+20+34+34+7=134 逻辑技能

## 一、技术债务（代码层）

### T1. Data Agent Engine 全部返回硬编码 Mock 数据（严重）

`main_project_lute/engine/data_processor.py` 是整个 Data Agent 的数据处理核心，但其 12 个处理方法**全部返回硬编码字典**。没有任何真实数据读取、数据库连接或计算逻辑。调用流程像是真的在执行——打印日志、路由 Skill、解析参数——但最终产出是预设的赝品。

涉及的 Skill：margin-attribution、cost-structure-analysis、voc-insights、contribution-calculation、channel-effect-analysis、ad-attribution、basket-analysis、dupont-analysis、cohort-analysis、social-media-analysis、scm-cost-anomaly-diagnosis、scm-inventory-health-diagnosis、scm-executive-summary。

SCM 三个任务（成本异常、库存健康、管理层摘要）尤为危险——它们返回的是 "Grey 数据状态" 占位符而非实际分析结果，但调用者无法从返回值中区分这是真实数据还是占位符。

### T2. sys.path.insert 污染（~13 处）

`main_project_lute/` 下的引擎、Agent、Output 模块普遍使用运行时 `sys.path.insert(0, str(PROJECT_ROOT))` 来拼凑 Python 路径。这有几个问题：
- `agent/core.py` 在文件内多次 `import sys`（第14行和第31行各一次）
- `engine/data_processor.py` 的 PROJECT_ROOT 计算是 `Path(__file__).resolve().parents[1].parent`，假设了从 engine/ 向上跳两级到仓库根——如果目录移动就会断裂
- 同仓库内模块导入应该用相对导入或安装为包，而不是靠字符串 hack

### T3. PPTGenerator 不生成真正的 PPTX（严重）

`main_project_lute/output/ppt_generator.py` 声称是 "PPT 生成器"，但 `_save_template()` 方法输出的是 JSON 文件（`output_path.write_text(json.dumps(...))`），而不是 .pptx。真正的 PPTX 生成在 `data_example/scripts/` 下用 python-pptx 直接操作。engine/output 模块与实际产出代码之间是完全脱节的。

### T4. ChartEngine 同样不产生真正的图表（严重）

`main_project_lute/output/chart_engine.py` 没有调用任何图表库。它是 engine/output 层的骨架，但实际图表在 `data_example/scripts/core/build_*.py` 中用 matplotlib/plotly 生成。

### T5. 测试基础设施不存在（严重）

- `tests/` 目录完全为空
- 9 个 Python 模块的文件头注释都写着 `python -m pytest agent/tests/test_*.py`、`python -m pytest engine/tests/test_*.py` 等，但这些 tests 目录一个都不存在
- 每个模块末尾有内联 `def test_*()` 函数，但都是硬编码演示逻辑，不包含任何 assertion
- 没有 CI/CD 配置文件、没有 tox.ini、没有 pyproject.toml

### T6. 无依赖管理文件

整个仓库没有 `requirements.txt`、`pyproject.toml`、`setup.py` 或 `Pipfile`。唯一相关的文件是 `scm/.agents/skills/fireworks-tech-graph/package.json`（属于第三方工具）。运行脚本需要的库（pandas、openpyxl、python-pptx、matplotlib、plotly、praw、playwright 等）零散地写在各个脚本的注释中，无法一次性安装。

### T7. 脚本组织混乱——core/ vs experimental/ vs sheet10/ vs standalone/

`data_example/scripts/` 下有四个子目录（core/ 13 脚本、experimental/ 14 脚本、sheet10/ 2 脚本、standalone/ 1 脚本），但分类标准不清晰：
- `run_专题02_双平台费率归因.py` 在 core/ 里，但 `run_phase2_topic2_pipeline.py` 在 experimental/ 里——同样是专题②，分散在两个目录
- core/ 脚本处理专题 01-05（Phase1 分析），experimental/ 处理 Phase2/Phase3，但 sheet10/ 又回到专题01——三个不同的脚本组织维度（按专题、按阶段、按Sheet编号）混在一起

### T8. "Basket Analysis" Skill 引用不存在的 SKILL.md

`data_processor.py` 中 `basket-analysis` 路由到 `_process_basket_analysis()`，`skills_router.py` 中将其别名映射到 `cbec-category-management`，但搜索 `skills/cross_border_ecommerce/skills/` 目录确认——没有 `basket-analysis/SKILL.md` 存在。这是一个死链接。

## 二、工程债务（架构层）

### E1. Agent/Engine/Output 层与实际执行完全脱节

Data Agent 架构定义了三层：
1. `agent/` — 意图解析 + Skill 路由
2. `engine/` — Skill 加载 + 数据处理 + 结果格式化
3. `output/` — PPT 生成 + 报告组装 + 图表引擎

但实际运行链路是直接调用 `data_example/scripts/` 下的独立脚本（如 `run_专题01_平台区域毛利归因.py` → `build_专题01_瀑布图汇报.py`），完全绕过 Agent/Engine/Output 三层。这意味着三层架构是理论设计、无法投入使用的纸面框架。

### E2. `.claude/skills/` 与 `.agents/skills/` 不完全一致

虽然 SKILLS_INVENTORY.md 声称二者"名称完全一致"，但 diff 扫描发现 7 个文件有差异：
- `ClaudeSkillsCreateSkills/SKILL.md`：产品名称不同（"Claude Code" vs "Codex"）
- `dispatching-parallel-agents/SKILL.md`
- `executing-plans/SKILL.md`
- `receiving-code-review/SKILL.md`
- `using-git-worktrees/SKILL.md`
等 7 个文件内容不一致。未来某个 Skill 在其中一边修改后，另一边不会同步——用户根据哪个入口进入会得到不同的指令。

### E3. SCM 模块通过 cbec- 前缀别名挂载到跨境电商 Skills 路由

`skills_router.py` 中将 SCM 任务路由到跨境电商 Skills：
```python
"scm-cost-anomaly-diagnosis": "cbec-cost-structure-analysis",
"scm-inventory-health-diagnosis": "cbec-inventory-optimization",
"scm-executive-summary": "cbec-data-reporting",
```
但 SCM 的成本异常诊断 != 跨境电商的成本结构分析，库存健康诊断 != 跨境电商的库存优化。这是为了凑路由而把不相关的 Skill 强行嫁接，会导致错误的分析模板被应用到错误的问题上。

### E4. `data_example/` 名不副实——它是实际的生产代码目录

名为 "data_example"（数据示例），但包含的是实际用于生成 Phase2/Phase3 PPT 产出的脚本。这个命名会误导新开发者，让他们以为这是示例代码可以忽略。

### E5. Phase2 和 Phase3 的脚本复用度为零

Phase2 和 Phase3 的 pipeline 脚本（`run_phase2_topic2_pipeline.py` vs `run_phase3_topic1_voc_pipeline.py` 等）结构相似但不共享代码。Mock 生成器（`generate_phase2_mock_data.py` vs `generate_phase3_topic1_voc_mock.py` 等）也分别独立实现，逻辑重复。

## 五、脆弱点债务

### V1. 仓库规模膨胀——2.8GB

`ref/` 目录占 1.5GB：
- `ref/aippt/` 1.1GB — PPT 模板和大文件
- `ref/books/` 321MB — PDF 书籍
- `scm/system_data/` 58MB

这些二进制大文件使 clone/stash/branch 操作缓慢，且 `ref/` 路径在 .gitignore 中未被排除。

### V2. .env 文件已被提交或可能被提交

`.env` 文件存在且包含 3 行（推测是 Biji API Key），虽然 `.gitignore` 中声明了 `.env`，但当前 git 状态显示该文件在工作区存在。如果一旦提交，API Key 将永久留在 git 历史中。

### V3. 硬编码路径散落 11+ 处

如果仓库路径再次变更（例如从 `ecom_ana_overview` 再迁移），11 个文件中硬编码的绝对路径将全部失效，且没有统一配置文件来管理路径。

### V4. 无错误处理机制

- `DataAgent.execute()` 的 except 块只是 `print(f"[错误] {e}")` 并返回 error 状态的 AgentResult，没有日志、没有告警、没有重试
- `SkillsRouter` 在 skills_index 加载失败时静默回退到关键词匹配，用户不知道自己在用降级路由
- 所有脚本使用 `subprocess.run()` 但部分没有检查 `returncode`

### V5. 关键数据源路径隐藏在脚本深处

数据文件路径（如 `专题一：分析数据总表.xlsx`）散布在各个 `run_专题*.py` 和 `build_专题*.py` 脚本中，没有集中的数据源配置。当 Excel 文件位置或 Sheet 名称变化时，需要逐脚本修改。

### V6. 项目路径变更脆弱性

从 `shopify_analysis` → `ecom_ana_overview` 的迁移已经引入 11 处断裂，且反映了两个核心问题：(a) 没有路径配置层，(b) 文档中的命令示例与实际路径不兼容。未来任何路径变更都将重现相同问题。

### V7. PDF/二进制依赖无版本锁定

`ref/books/` 中的 PDF 书籍作为分析参考，但没有记录这些文件的来源、版本或获取方式。如果丢失或损坏，无法从远处恢复。
