# 供应链履约看板指标体系与指标字典草稿

## 1. 事实

- Excel 有 3 个 sheet：`指标`、`需求`、`指标-副本`。
- `指标` sheet 抽取出 114 行指标/字段：P0 44、P1 23、P2 19、P3 28。
- `需求` sheet 固定了筛选项、指标速览、多维详情表和 10 个业务主题。
- 本地知识库已有 MECE 供应链指标体系，包含 178 个指标节点：L0 6、L1 8、L2 25、L3 139。
- Readdy 原型可见数值包括：原始总单量 128,456、拆分总单量 135,892、24h 审单及时率 94.8%、48h 审单及时率 98.2%、24h 商品发货及时率 82.5%、订单分配 133,000。
- 2026-06-25 业务口径已确认：缺货拆成订单缺货、库存缺货、预测缺货；妥投签收时间以 TMS 为准；平均发货成本 V0 只占位；未发货预警保留独立模块；审核效能区分系统自动审核与人工审核。

## 2. 指标体系分层

| 层级 | 本看板定义 | 与知识库关系 |
|---|---|---|
| L0 | 履约体验达成率 | 知识库已有 `fulfillment_experience_rate`，可作为本看板北极星 |
| L1 | 订单履约时效、物流妥投、仓储执行、异常闭环 | 对应知识库的“物流运输与履约体验”“仓储执行与库存准确”“逆向异常与售后闭环” |
| L2 | 订单规模、审单及时、审核发货、付款发货、商品发货、妥投、跨区、缺货 | Excel sheet 2 的业务指标层 |
| L3 | 原始总单量、拆分总单量、24h/48h 及时率、3/5/10 天妥投率、平均间隔、明细字段 | 直接进入指标字典和图表口径 |

## 3. Excel 与知识库的吻合点

| 吻合点 | 说明 |
|---|---|
| 履约体验达成 | 知识库已有履约体验达成率，Excel 将其拆成付款、审核、发货、签收四段，适合形成时效链路 |
| 数据治理要求 | 知识库强调 source coverage、血缘、证据等级；Excel 已提供来源表、字段、待确认、Owner |
| 仓库与物流维度 | 两边都要求仓库、物流渠道、国家/区域作为核心切片 |
| 商品维度 | 知识库强调 SKU/库存对象，Excel 强调供应链 SKU、SPU、品线、品类、GTM 组 |
| 异常闭环 | 知识库有风险与行动闭环，Excel 提供订单异常标识和问题件明细入口 |

## 4. 差异

| 差异 | 知识库 | Excel/业务需求 | 处理建议 |
|---|---|---|---|
| 粒度 | 企业级 SCM 指标，覆盖采购、库存、成本、治理 | 订单级履约运营，强调单号、SKU、审核人、发货、签收 | 看板 MVP 以订单/商品/包裹粒度为主，保留与 L0/L1 的映射 |
| 成本域 | 成本财务与资金效率是独立大域 | 只出现平均发货成本，且来源待补 | 平均发货成本 V0 只占位，不进入核心 KPI |
| 缺货域 | 知识库可从库存、覆盖、风险阈值解释 | Excel 写明缺货分析需重新明确 | 按订单缺货、库存缺货、预测缺货三类建指标 |
| 数据源状态 | 多数为 draft governance/source coverage | Excel 提供部分 ERP 表字段，TMS/成本/审核日志仍需接入 | TMS 作为签收权威源；成本占位；审核日志需补审核方式 |
| 未发货预警 | 知识库可作为风险雷达 | Excel 标注 0622 会议确认删除模块 | 按当前业务决策保留独立模块 |

## 5. P0 指标字典

| 指标 | 定义 | 分母/粒度 | 来源/字段 | 当前状态 |
|---|---|---|---|---|
| 原始总单量 | 同一拆单分析群体内的拆分前有效平台订单量：已付款、进入有效自发货范围，排除已付款取消与拆单废弃 | 平台单号 `ref_no` 去重 | `lute_os.t_erp_order.ref_no` + `paid_flag`, `valid_self_fulfillment_flag`, `paid_cancelled_flag`, `split_abandoned_flag` | 可进入 V0 |
| 拆分总单量 | 与原始总单量同一群体的拆分后有效自发货订单量；排除拆单废弃和已付款取消 | 系统自发货单号 `erp_code` 去重 | `lute_os.t_erp_order.erp_code` + 与分母相同的归一化状态标记 | 可进入 V0 |
| 未审核单量 | 已付款未审核订单数量，异常状态订单也要展示 | 拆分总单量 | `order_status` + 审核状态 | 待确认状态码 |
| 异常订单量 | 订单异常小状态数量 | 拆分总单量 | `order_status` 小状态映射 | 可进入 V0 |
| 24h/48h 审单及时率 | 末次审核时间 - 付款时间小于阈值的单量 / 总有效订单 | 拆分总单量 | `pay_time`, `os_auth_time` | 多次审核日志待优化 |
| 24h/48h 审核发货及时率 | 发货时间 - 审核时间小于阈值的单量 / 已审核单量 | 已审核单量 | `os_auth_time`, `send_time` | 可进入 V0 |
| 24h/48h 付款发货及时率 | 发货时间 - 付款时间小于阈值的单量 / 总单量 | 拆分总单量 | `pay_time`, `send_time` | 可进入 V0 |
| 24h/48h 商品发货及时率 | 发货时间 - 付款时间小于阈值的商品数 / 总订单商品数量 | 供应链 SKU 商品行 | `t_erp_order_item.sku_code`, `num` | 可进入 V0 |
| 超期未发货率 | 截至当日，付款时间大于 7 天仍未发货的订单 / 总订单数 | 拆分总单量 | `pay_time`, `send_time`, `order_status` | 可进入 V0 |
| 未发货预警 | 付款后超过 SLA 仍未发货的订单，按 7/10/15 天分层 | 拆分总单量 | `pay_time`, `send_time`, `order_status`, `warehouse_code`, `sku_code` | 独立模块进入 V0 |
| 3/5/10 天妥投率 | TMS 签收时间 - 发货时间小于阈值的包裹数 / 总发货包裹数 | 包裹/跟踪号 | `track_order_code`, `tms_sign_time` | TMS 为准 |
| 累计妥投率 | 累计签收订单包裹数 / 总发货包裹数 | 包裹/跟踪号 | TMS `package_status`, `tms_sign_time` | TMS 为准 |
| 超期未妥投率 | 发货时间大于 12 天仍未签收包裹数 / 总发货包裹数 | 包裹/跟踪号 | `send_time`, `tms_sign_time` | TMS 为准 |
| 拆单率 | `拆分后有效自发货订单量 / NULLIF(拆分前有效平台订单量, 0) - 1`；两端使用同一订单群体，分母为 0 时返回 `null` | 原始单号与拆分单号映射 | `ref_no`, `erp_code` + 同群体状态过滤 | 可进入 V0 |
| 未发货率 | 未发货订单 / 总单量 | 拆分总单量 | `order_status`, `send_time` | 可进入 V0 |
| 异常订单率 | 异常订单量 / 总单量 | 拆分总单量 | `order_status` 小状态 | 可进入 V0 |
| 跨区发货率 | 目的国家/地区与实际发货仓所在国家/地区不一致 | 订单 + 发货仓 | `order_receive.country`, `warehouse.country_code` | 可进入 V0 |
| 预售单比率 | 预售订单单量 / 总单量 | 原始平台单 | `t_erp_order_label.label_name` | 需固定标签规则 |
| 平均发货成本 | V0 占位：已发货成本 / 已发货单量，暂不作为核心指标 | 已发货订单 | 物流商后台 | 占位，不进核心 KPI |
| 审核效能 | 系统自动审核与人工审核分开统计；人工审核按审核人看单量与时效 | 审核方式 + 审核人 + 审核事件 | `audit_actor_type`, `audit_user`, `audit_time` | 需补审核日志 |
| 标发失败量 | 标发失败订单数量 | 自发货订单 | `mark_shipped_status` | 可进入 V0 |

## 6. 已确认口径与剩余数据依赖

| 项目 | 已确认口径 | 剩余数据依赖 |
|---|---|---|
| 缺货分析 | 订单缺货、库存缺货、预测缺货三类 | 订单异常、可售库存、在途、需求预测、覆盖天数 |
| 签收时间来源 | 妥投签收时间以 TMS 为准 | TMS `sign_time`、包裹状态和同步频率 |
| 发货成本 | 平均发货成本 V0 只占位，不作为核心指标 | 物流成本宽表或物流商后台成本字段 |
| 审核效能 | 系统自动审核与人工审核必须区分 | 审核事件表、审核方式、审核人、审核时间 |
| 未发货预警 | 保留独立模块 | 超期分层、责任域、Owner、下一步动作 |
| 仓库类型映射 | 用仓库维表固定小包仓、海外仓、平台仓等类型 | 仓库主数据与类型映射 |
| 预售标签 | 预售订单需单独标识 | 预售标签命名和筛选规则 |

## 7. 结论

事实：当前 Excel 已足够支撑履约看板 V0 的指标体系和大部分页面原型；知识库提供上层 MECE 框架；Readdy 原型提供样式和样例数值；5 条业务口径已在本草稿中固化。

推断：V0 应先定位为订单履约运营看板，而不是完整供应链经营驾驶舱。采购、库存、成本等知识库能力只作为关联解释，不进入本轮主页面。

不确定项：TMS、库存、预测、物流成本和审核日志的真实字段、同步频率、数据质量仍需数据 owner 在建模阶段确认；本文件仍是草稿，不代表生产 SQL 已完成。
